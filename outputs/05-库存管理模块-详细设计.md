# 库存管理模块（Inventory Management）详细设计

> 版本：v1.0  
> 日期：2026-04-30  
> 适用行业：服装生产销售  
> 前置依赖：基础数据模块（SKU、仓库/库位、物料）、订单管理、采购管理

---

## 一、模块总览

库存是整个系统的交汇点 — 采购入库、生产入库、销售出库、领料出库，所有业务最终都落在这里。库存管理模块做不好，整个系统的数据可信度就崩了。

服装行业库存的特殊性：

1. **SKU粒度极细** — 一个款3色5码 = 15个SKU，每个SKU独立计数
2. **批次追溯是刚需** — 同款同色必须同批面料，成品也要按批次追溯
3. **季节性库存贬值极快** — 过季库存几乎等于废品，库龄分析至关重要
4. **多仓多库位** — 原料仓、成品仓、退货仓，每个仓有存储位/拣货位/暂存位
5. **成品与原料库存模型不同** — 成品按SKU管理，面料按批次+米数管理，辅料按批次+件数管理

本模块包含：
1. 库存状态模型 — 四类库存与状态转换
2. 库存流水与变更日志 — 每次变动可追溯
3. 库存预留与释放 — 销售订单锁定库存
4. 库存操作 — 入库、出库、调拨、调整
5. 库存盘点 — 明盘/盲盘
6. 库存预警与报表

## 二、库存状态模型

### 2.1 为什么需要四类库存？

直接用一个"库存数量"字段管理库存，在多人并发操作时会出问题：

```
场景：仓库里有100件黑色M码风衣

9:00  销售订单A需要60件 → 查库存=100 → 够 → 准备出库
9:01  销售订单B需要50件 → 查库存=100 → 够 → 准备出库
9:05  两个出库单同时执行 → 实际出库110件 → 库存变成-10 ❌
```

解决方案：把库存分成四个池子：

```
┌─────────────────────────────────────────────┐
│              实际库存（On-hand）              │
│  仓库里实实在在存在的货物总数                  │
│                                              │
│  ┌──────────────┐  ┌──────────────┐         │
│  │   可用库存    │  │   锁定库存    │         │
│  │  Available    │  │   Locked     │         │
│  │  可自由分配   │  │  已预留未出库  │         │
│  └──────────────┘  └──────────────┘         │
│                                              │
│  ┌──────────────┐  ┌──────────────┐         │
│  │   质检库存    │  │   在途库存    │         │
│  │   QC Hold     │  │   In-Transit  │         │
│  │  到货待检验   │  │  已采购未到货  │         │
│  └──────────────┘  └──────────────┘         │
└─────────────────────────────────────────────┘

关系：实际库存 = 可用库存 + 锁定库存 + 质检库存
在途库存不在仓库里，是"即将到达"的数量
```

**各类库存的含义：**

| 库存类型 | 说明 | 何时增加 | 何时减少 |
|---------|------|---------|---------|
| 可用库存（Available） | 可以自由分配给新订单的库存 | 入库检验合格、锁定释放、盘点盈余 | 锁定预留、出库扣减、盘点亏损 |
| 锁定库存（Locked） | 已分配给销售订单但尚未出库的库存 | 销售订单确认时预留 | 出库完成、订单取消释放 |
| 质检库存（QC） | 已到货但尚未完成检验的库存 | 采购到货收货 | 检验合格→可用、检验不合格→退货 |
| 在途库存（In-Transit） | 已下采购订单但尚未到货的数量 | 采购订单下发给供应商 | 到货收货 |

### 2.2 库存状态转换

```
                        采购订单下发
                            │
                            ▼
                    ┌──────────────┐
                    │   在途库存    │
                    └──────┬───────┘
                           │ 到货收货
                           ▼
                    ┌──────────────┐
                    │   质检库存    │
                    └──────┬───────┘
                      ┌────┴────┐
                      │检验合格  │检验不合格
                      ▼         ▼
               ┌──────────┐  退货出库（质检库存减少）
               │ 可用库存  │
               └─────┬────┘
                ┌────┴────┐
                │销售预留  │
                ▼         │
         ┌──────────┐    │
         │ 锁定库存  │    │
         └─────┬────┘    │
           ┌───┴───┐     │
           │出库完成│取消  │
           ▼       ▼     │
      出库扣减    释放回  │
                 可用库存  │
                         │
                    生产入库│
                  (成品完工)│
                         │
                         ▼
                    ┌──────────┐
                    │ 可用库存  │
                    └──────────┘
```

### 2.3 关键约束

- **可用库存永远不能为负** — 扣减前必须校验，库存不足时拒绝操作
- **锁定库存 ≤ 实际库存** — 锁定的是真实存在的货物
- **质检库存和锁定库存互斥** — 同一批货物不能同时在检又被锁定
- **在途库存不占仓库物理空间** — 只是数量跟踪，不等同于实际库存
- **所有库存变动必须走操作流水** — 不能直接 UPDATE 数量字段

## 三、库存数据表设计

### 3.1 成品库存

```
inventory_sku（成品库存 — 按SKU+仓库+批次）
├── id                       BIGPK          主键
├── sku_id                   BIGINT         SKU ID（外键→md_sku）
├── warehouse_id             BIGINT         仓库ID
├── location_id              BIGINT         库位ID（可选，精细管理时必填）
├── batch_no                 VARCHAR(32)    批次号（生产批次或入库批次）
├── available_qty            INTEGER        可用数量
├── locked_qty               INTEGER        锁定数量
├── qc_qty                   INTEGER        质检数量
├── total_qty                INTEGER        实际库存 = available + locked + qc
├── in_transit_qty           INTEGER        在途数量（冗余，方便查询；权威数据在 inventory_in_transit 表，日终对账校验一致性）
├── unit_cost                DECIMAL(12,2)  单位成本（加权平均法）
├── last_inbound_date        DATE           最后入库日期
├── last_outbound_date       DATE           最后出库日期
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

唯一约束：(sku_id, warehouse_id, batch_no) — 同SKU同仓库同批次一行记录
```

**为什么 batch_no 是唯一约束的一部分？**

同一SKU同一仓库可能有多个批次的库存。不同批次不能混着管理，因为：
- 生产追溯需要按批次回查
- 库龄分析需要按批次计算
- 先进先出（FIFO）出库需要按批次排序

### 3.2 原料库存

```
inventory_material（原料库存 — 按物料+仓库+批次）
├── id                       BIGPK          主键
├── material_id              BIGINT         物料ID（外键→md_material）
├── warehouse_id             BIGINT         仓库ID
├── location_id              BIGINT         库位ID
├── batch_no                 VARCHAR(32)    批次号 ★关键★
├── supplier_id              BIGINT         供应商ID（面料来自哪个供应商）
├── procurement_order_id     BIGINT         采购订单ID（来自哪次采购）
├── available_qty            DECIMAL(12,2)  可用数量（面料按米，辅料按个）
├── locked_qty               DECIMAL(12,2)  锁定数量
├── qc_qty                   DECIMAL(12,2)  质检数量
├── total_qty                DECIMAL(12,2)  实际库存
├── in_transit_qty           DECIMAL(12,2)  在途数量（冗余，方便查询；权威数据在 inventory_in_transit 表，日终对账校验一致性）
├── unit_cost                DECIMAL(12,2)  单位成本
├── roll_count               INTEGER        卷数（面料专用，1卷约60-100米）
├── last_inbound_date        DATE
├── last_outbound_date       DATE
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

唯一约束：(material_id, warehouse_id, batch_no)
```

**面料为什么需要 roll_count？** 因为面料是按卷存放的，一卷可能是60米或100米。库存显示"2400米"时，仓库管理员需要知道这是24卷还是40卷，因为卷数影响存储空间和拣货效率。

### 3.3 在途库存

```
inventory_in_transit（在途库存 — 按采购订单行跟踪）
├── id                       BIGPK          主键
├── procurement_order_id     BIGINT         采购订单ID
├── procurement_line_id      BIGINT         采购订单行ID
├── material_id              BIGINT         物料ID（成品或原料）
├── warehouse_id             BIGINT         目标仓库ID
├── expected_qty             DECIMAL(12,2)  预计到货数量
├── received_qty             DECIMAL(12,2)  已收货数量
├── remaining_qty            DECIMAL(12,2)  剩余在途 = expected - received
├── expected_arrival_date    DATE           预计到货日期
├── status                   VARCHAR(16)    PENDING/PARTIAL_RECEIVED/FULLY_RECEIVED
├── created_by / created_at / updated_by / updated_at / deleted / version
└──
```

### 3.4 库存汇总视图

为了高效查询"某个SKU在所有仓库的库存情况"，提供汇总视图：

```sql
-- 成品库存汇总视图（按SKU）
CREATE VIEW v_inventory_sku_summary AS
SELECT 
    sku_id,
    SUM(available_qty)  AS total_available,
    SUM(locked_qty)     AS total_locked,
    SUM(qc_qty)         AS total_qc,
    SUM(total_qty)      AS total_onhand,
    SUM(in_transit_qty) AS total_in_transit
FROM inventory_sku
WHERE deleted = false
GROUP BY sku_id;
```

## 四、库存流水与变更日志

### 4.1 设计原则

**库存数量不能直接 UPDATE，必须通过"操作+流水"变更。**

这是企业级库存系统的铁律。任何直接改数字的操作都是bug的温床。正确的做法是：

```
每次库存变更 = 写一条操作记录 + 根据操作类型更新库存字段
```

### 4.2 库存操作类型

| 操作类型 | 影响的库存字段 | 说明 |
|---------|--------------|------|
| INBOUND_PURCHASE | qc_qty += N | 采购到货，进入质检 |
| INBOUND_PRODUCTION | available_qty += N | 生产完工入库，直接可用 |
| INBOUND_RETURN | qc_qty += N | 客户退货，进入质检 |
| QC_PASS | available_qty += N, qc_qty -= N | 质检合格，转入可用 |
| QC_FAIL | qc_qty -= N | 质检不合格，退货出库 |
| ALLOCATE | available_qty -= N, locked_qty += N | 锁定预留 |
| RELEASE | available_qty += N, locked_qty -= N | 释放预留 |
| OUTBOUND_SALES | locked_qty -= N | 销售出库，从锁定扣减 |
| OUTBOUND_MATERIAL | available_qty -= N | 领料出库，从可用扣减 |
| OUTBOUND_TRANSFER | available_qty -= N（源仓）, available_qty += N（目标仓）| 调拨出库+入库 |
| ADJUST_GAIN | available_qty += N | 盘点盈余/手工调增 |
| ADJUST_LOSS | available_qty -= N | 盘点亏损/手工调减 |

### 4.3 库存操作记录表

```
inventory_operation（库存操作记录 — 每次库存变更一行）
├── id                       BIGPK          主键
├── operation_no             VARCHAR(32)    操作单号（编码规则生成）
├── operation_type           VARCHAR(32)    操作类型（上表中的枚举）
├── inventory_type           VARCHAR(16)    库存类型：SKU/MATERIAL
├── inventory_id             BIGINT         库存记录ID（inventory_sku 或 inventory_material 的 ID）
├── sku_id                   BIGINT         SKU ID（成品时使用）
├── material_id              BIGINT         物料ID（原料时使用）
├── warehouse_id             BIGINT         仓库ID
├── location_id              BIGINT         库位ID
├── batch_no                 VARCHAR(32)    批次号
├── quantity                 DECIMAL(12,2)  操作数量（正数）
├── available_before         DECIMAL(12,2)  操作前可用数量
├── available_after          DECIMAL(12,2)  操作后可用数量
├── locked_before            DECIMAL(12,2)  操作前锁定数量
├── locked_after             DECIMAL(12,2)  操作后锁定数量
├── qc_before                DECIMAL(12,2)  操作前质检数量
├── qc_after                 DECIMAL(12,2)  操作后质检数量
├── total_before             DECIMAL(12,2)  操作前实际库存
├── total_after              DECIMAL(12,2)  操作后实际库存
├── source_type              VARCHAR(32)    来源单据类型
│                                          SALES_ORDER / PRODUCTION_ORDER / PROCUREMENT_ORDER /
│                                          OUTBOUND_ORDER / INBOUND_ORDER / STOCKTAKING / MANUAL
├── source_id                BIGINT         来源单据ID
├── source_no                VARCHAR(32)    来源单据编号
├── unit_cost                DECIMAL(12,2)  本次操作的单位成本
├── cost_amount              DECIMAL(12,2)  本次操作的成本金额 = quantity × unit_cost
├── operator_id              BIGINT         操作人ID
├── operated_at              TIMESTAMP      操作时间
├── remark                   TEXT           备注
└── created_at               TIMESTAMP      记录创建时间（不可修改）
```

**分区策略（inventory_operation 按月分区）：**

由于 `inventory_operation` 表写入频率极高（每次库存变更必写一行），长期运行后数据量巨大。采用 PostgreSQL 原生分区表按月分区：

```sql
CREATE TABLE inventory_operation (
    id              BIGSERIAL,
    operation_no    VARCHAR(32),
    operation_type  VARCHAR(32),
    -- ... 其他字段同上 ...
    operated_at     TIMESTAMP NOT NULL
) PARTITION BY RANGE (operated_at);

-- 每月一个分区（可用 pg_partman 扩展自动管理）
CREATE TABLE inventory_operation_2026_01 PARTITION OF inventory_operation
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE inventory_operation_2026_02 PARTITION OF inventory_operation
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
-- ...
```

**归档策略：**

- 0-12 个月：热数据，保留在主库分区中，正常查询
- 12-36 个月：温数据，DETACH 分区后迁移到归档库，查询走联邦查询（postgres_fdw）
- 36 个月以上：冷数据，导出为 Parquet/CSV 存对象存储，仅审计时导入查看

**关键索引：**

```sql
-- 高频查询：按SKU+仓库+时间查操作流水
CREATE INDEX idx_inv_op_sku_wh_time ON inventory_operation (sku_id, warehouse_id, operated_at);
CREATE INDEX idx_inv_op_material_wh_time ON inventory_operation (material_id, warehouse_id, operated_at);
-- 按来源单据反查
CREATE INDEX idx_inv_op_source ON inventory_operation (source_type, source_id);
-- 按操作类型+时间筛选（报表用）
CREATE INDEX idx_inv_op_type_time ON inventory_operation (operation_type, operated_at);
```

### 4.4 为什么记录 before/after？

**可追溯 + 可校验 + 可对账。**

- 追溯：任何时刻的库存状态都可以通过流水重建
- 校验：`after - before` 必须等于 `quantity`（根据操作类型取正负），数据损坏时可以发现
- 对账：日终/月终时，汇总所有操作的数量变动，必须等于库存表当天/当月的变动量

### 4.5 库存变更的核心Service方法

```java
@Service
public class InventoryDomainService {

    /**
     * 通用库存变更方法 — 所有库存操作必须通过此方法
     * 禁止直接 UPDATE inventory_sku / inventory_material 的数量字段
     */
    @Transactional
    public void changeInventory(InventoryChangeRequest request) {
        // 1. 加载库存记录（乐观锁）
        InventoryRecord record = loadInventoryRecord(request);
        
        // 2. 记录变更前快照
        InventorySnapshot before = record.snapshot();
        
        // 3. 校验操作合法性
        validateChange(request, record);
        
        // 4. 执行变更
        applyChange(record, request);
        
        // 5. 记录变更后快照
        InventorySnapshot after = record.snapshot();
        
        // 6. 保存库存记录（乐观锁保证并发安全）
        saveInventoryRecord(record);
        
        // 7. 写操作流水
        InventoryOperation operation = buildOperation(request, before, after);
        operationRepository.insert(operation);
        
        // 8. 发布库存变更领域事件
        publishInventoryChangedEvent(request, before, after);
    }

    /**
     * 校验逻辑
     */
    private void validateChange(InventoryChangeRequest request, InventoryRecord record) {
        switch (request.getOperationType()) {
            case ALLOCATE:
                // 锁定数量不能超过可用库存
                if (record.getAvailableQty() < request.getQuantity()) {
                    throw new InsufficientInventoryException(
                        record.getSkuId(), record.getWarehouseId(),
                        record.getAvailableQty(), request.getQuantity());
                }
                break;
            case OUTBOUND_SALES:
                // 出库数量不能超过锁定库存
                if (record.getLockedQty() < request.getQuantity()) {
                    throw new BizException("锁定库存不足，无法出库");
                }
                break;
            case OUTBOUND_MATERIAL:
                // 领料数量不能超过可用库存
                if (record.getAvailableQty() < request.getQuantity()) {
                    throw new InsufficientInventoryException(...);
                }
                break;
            case QC_PASS:
                // 合格数量不能超过质检库存
                if (record.getQcQty() < request.getQuantity()) {
                    throw new BizException("质检库存不足");
                }
                break;
            case ADJUST_LOSS:
                // 亏损数量不能超过可用库存
                if (record.getAvailableQty() < request.getQuantity()) {
                    throw new BizException("可用库存不足，无法调减");
                }
                break;
            // INBOUND_* 不需要校验上限
        }
    }

    /**
     * 执行变更 — 根据操作类型更新对应字段
     */
    private void applyChange(InventoryRecord record, InventoryChangeRequest request) {
        BigDecimal qty = request.getQuantity();
        
        switch (request.getOperationType()) {
            case INBOUND_PURCHASE:
                record.setQcQty(record.getQcQty().add(qty));
                break;
            case INBOUND_PRODUCTION:
                record.setAvailableQty(record.getAvailableQty().add(qty));
                break;
            case QC_PASS:
                record.setQcQty(record.getQcQty().subtract(qty));
                record.setAvailableQty(record.getAvailableQty().add(qty));
                break;
            case QC_FAIL:
                record.setQcQty(record.getQcQty().subtract(qty));
                break;
            case ALLOCATE:
                record.setAvailableQty(record.getAvailableQty().subtract(qty));
                record.setLockedQty(record.getLockedQty().add(qty));
                break;
            case RELEASE:
                record.setAvailableQty(record.getAvailableQty().add(qty));
                record.setLockedQty(record.getLockedQty().subtract(qty));
                break;
            case OUTBOUND_SALES:
                record.setLockedQty(record.getLockedQty().subtract(qty));
                break;
            case OUTBOUND_MATERIAL:
                record.setAvailableQty(record.getAvailableQty().subtract(qty));
                break;
            case ADJUST_GAIN:
                record.setAvailableQty(record.getAvailableQty().add(qty));
                break;
            case ADJUST_LOSS:
                record.setAvailableQty(record.getAvailableQty().subtract(qty));
                break;
        }
        
        // 重新计算实际库存
        record.setTotalQty(
            record.getAvailableQty()
                .add(record.getLockedQty())
                .add(record.getQcQty())
        );
    }
}
```

### 4.6 并发安全

库存扣减是典型的高并发场景，必须保证不超卖：

**方案一：乐观锁（推荐）**

```java
// inventory_sku / inventory_material 表的 version 字段
UPDATE inventory_sku 
SET available_qty = ?, locked_qty = ?, total_qty = ?, version = version + 1
WHERE id = ? AND version = ?;

// 如果影响行数为0，说明被其他事务修改了，重试
```

适合场景：中等并发，冲突概率不高。

**方案二：Redis分布式锁 + 扣减（高并发时）**

```java
// 针对热门SKU，用Redis做预扣减
String lockKey = "inventory:lock:" + skuId + ":" + warehouseId;
RLock lock = redissonClient.getLock(lockKey);
try {
    lock.lock(5, TimeUnit.SECONDS);
    // 执行库存变更
} finally {
    lock.unlock();
}
```

**方案三：数据库行级锁（最简单）**

```sql
SELECT * FROM inventory_sku 
WHERE sku_id = ? AND warehouse_id = ? AND batch_no = ? 
FOR UPDATE;
```

适合场景：并发量不大，简单可靠。

**本项目建议：** 初期用乐观锁（version字段），如果某个热点SKU出现频繁冲突，针对该SKU切换到Redis锁。不需要全局都用分布式锁。

### 5.2 预留策略

**成品预留策略：**

```
销售订单确认时 → 触发库存预留
    │
    ├── 有足够可用库存 → 全额预留 → 状态 ACTIVE
    ├── 部分可用 → 预留可用部分 → 状态 ACTIVE，记录缺口
    └── 完全无库存 → 不预留 → 记录缺货
    
预留后：
    可用库存 -= 预留数量
    锁定库存 += 预留数量
```

**原料预留策略（领料出库场景）：**

```
生产订单下达时 → 触发原料预留
    │
    ├── 按BOM计算各原料需求
    ├── 检查各原料可用库存
    ├── 有库存 → 预留（锁定可用库存）
    └── 不足 → 记录缺口（通知采购）
    
注意面料的批次约束：
    同一生产订单行（同一颜色款）需要的面料
    必须从同一个批次预留
    如果单批次不够 → 尽量从同批次多预留
    实在不行 → 标记"需分批领料"，仓库分批备货
```

### 5.3 预留的批量分配算法

当多个销售订单争抢同一SKU的库存时，需要公平分配：

```java
/**
 * 库存分配算法 — 按订单优先级分配
 */
public List<AllocationResult> allocate(AllocationRequest request) {
    // 1. 查询该SKU在所有仓库的可用库存
    List<InventorySku> inventories = inventoryRepo.findBySkuId(request.getSkuId());
    BigDecimal totalAvailable = inventories.stream()
        .map(InventorySku::getAvailableQty)
        .reduce(BigDecimal.ZERO, BigDecimal::add);
    
    // 2. 查询所有待分配的预留请求（按优先级排序：客户等级 > 订单日期）
    List<AllocationRequest> pendingRequests = allocationRepo
        .findPendingBySkuId(request.getSkuId());
    
    // 3. 逐个分配
    BigDecimal remainingAvailable = totalAvailable;
    List<AllocationResult> results = new ArrayList<>();
    
    for (AllocationRequest req : pendingRequests) {
        if (remainingAvailable.compareTo(BigDecimal.ZERO) <= 0) {
            // 库存耗尽，剩余请求标记为缺货
            results.add(new AllocationResult(req, BigDecimal.ZERO, req.getDemandQty()));
            continue;
        }
        
        BigDecimal allocated = remainingAvailable.min(req.getDemandQty());
        BigDecimal shortfall = req.getDemandQty().subtract(allocated);
        
        results.add(new AllocationResult(req, allocated, shortfall));
        remainingAvailable = remainingAvailable.subtract(allocated);
    }
    
    return results;
}
```

### 5.4 预留过期自动释放

```
inventory_allocation 的 expire_at 字段：

预留时设置过期时间（如7天）
    │
    ▼
定时任务每30分钟扫描
    │
    ├── 已过期且状态=ACTIVE → 自动释放
    │     locked_qty -= remaining_qty
    │     available_qty += remaining_qty
    │     allocation.status = EXPIRED
    │     写库存操作流水（RELEASE）
    │
    └── 未过期 → 跳过
```

## 七、库位推荐

### 7.1 入库库位推荐

入库时系统推荐库位，减少仓库人员找位子的时间：

**推荐规则（优先级从高到低）：**

1. **同SKU已有库存的库位** — 同一批货物放在一起，方便管理
2. **同类物料的空闲库位** — 面料放面料区，辅料放辅料区
3. **同区域的空闲库位** — 就近原则
4. **任意空闲库位**

```java
public Location recommendInboundLocation(Long warehouseId, Long skuId, Long materialId) {
    // 规则1：同SKU已有库存的库位
    List<Location> sameSkuLocations = locationRepo.findLocationsWithInventory(
        warehouseId, skuId, materialId);
    if (!sameSkuLocations.isEmpty()) {
        return sameSkuLocations.get(0); // 取第一个有空间的
    }
    
    // 规则2：同类物料的空闲存储位
    String zoneType = determineZoneType(skuId, materialId);
    List<Location> emptyLocations = locationRepo.findEmptyLocations(
        warehouseId, zoneType, "STORAGE");
    if (!emptyLocations.isEmpty()) {
        return emptyLocations.get(0);
    }
    
    // 规则3：同区域任意空闲库位
    // 规则4：任意空闲库位
    // ...
    
    throw new BizException("仓库无空闲库位");
}
```

### 7.2 出库拣货推荐（FIFO + 批次优先）

出库时推荐从哪个库位、哪个批次取货：

**推荐规则：**

1. **批次优先** — 同款同色优先从同一批次取（服装行业硬约束）
2. **先进先出（FIFO）** — 同批次内先入库的先出
3. **拣货位优先** — 优先从拣货位出，不够再从存储位补

```
需要出库：黑色M码风衣 50件

查找顺序：
1. 查找黑色M码在当前仓库的所有库存记录（按批次排序）
2. 批次B20260401（入库早）→ 有30件在拣货位A-01-02 → 推荐先取30件
3. 批次B20260401 → 有20件在存储位A-02-01 → 推荐再取20件
4. 批次B20260415（入库晚）→ 有100件 → 不推荐（FIFO原则后取）

如果必须跨批次（单批次不够）：
5. 批次B20260401 → 取30件
6. 批次B20260415 → 取20件 → 标记"分批出库"
```

## 八、库存盘点

### 8.1 盘点类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| 全面盘点 | 仓库所有库存逐项盘点 | 年终/季末 |
| 循环盘点 | 每天盘点一部分，循环覆盖 | 日常轮换 |
| 抽盘 | 随机抽取部分SKU盘点 | 不定期抽查 |
| 动盘 | 有出入库操作的SKU当天盘点 | 高价值物料 |

### 8.2 明盘与盲盘

**明盘：** 盘点时可以看到系统库存数量，盘点人对照填写实际数量。

**盲盘：** 盘点时看不到系统库存数量，只填写实际数量，提交后系统自动比对差异。

**推荐用盲盘。** 明盘时盘点人员容易受系统数量影响（"系统说100个，我就数到100个差不多了"），盲盘更能发现真实差异。

### 8.4 盘点流程

```
创建盘点单（选择仓库/库区/盘点类型/明盘或盲盘）
    │
    ▼
生成盘点行（系统根据选择范围自动生成待盘SKU/物料列表）
    │
    ▼
开始盘点（状态：IN_PROGRESS）
    │  盘点人逐项填写实盘数量
    │  盲盘模式：看不到系统数量
    │  明盘模式：可看到系统数量作参考
    ▼
提交盘点结果
    │
    ▼
差异比对（状态：DIFF_REVIEW）
    │  系统自动计算：diff_qty = actual_qty - system_qty
    │  只展示有差异的行
    ▼
差异审核
    │  仓库主管审核差异
    │  差异大的行 → 安排复盘（第二轮盘点）
    │  差异小的行 → 直接确认
    ▼
库存调整（状态：COMPLETED）
    │  对确认的差弧行执行库存调整
    │  盘盈 → ADJUST_GAIN（available_qty += diff_qty）
    │  盘亏 → ADJUST_LOSS（available_qty -= diff_qty）
    │  每次调整都写库存操作流水
    ▼
完成
```

### 8.5 盘点期间的库存冻结

**关键规则：盘点期间，被盘点的库位/SKU禁止出入库操作。**

否则会出现：盘点人刚数完100件，仓库又入库了50件，盘点结果就不准了。

```
盘点单创建时：
    为涉及的库位设置"盘点冻结"标记
    
其他出入库操作校验时：
    if (location.isFrozen()) {
        throw new BizException("该库位正在盘点，禁止出入库操作");
    }

盘点完成时：
    解除库位冻结标记
```

**实现方式：** 在 `md_location` 表使用 `FROZEN` 状态值实现（status = 'FROZEN'）。本项目并发量小（<20人），数据库字段方案简单可靠，与盘点操作同事务，不存在缓存与数据库不一致的风险。

## 十、库存成本核算

### 10.1 成本流转链路

采用简单的成本流转：面料采购价 → 领料出库 → 生产入库 → 成品库存成本。不包括人工成本分摊、制造费用分摊（由财务系统处理）。

```
采购入库 → 面料单位成本 = 采购单价（含税）
    │
    ▼
领料出库 → 领料成本 = 面料单位成本 × 领料数量
    │
    ▼
生产入库 → 成品单位成本 = Σ(领料成本) / 完工数量
    │           （加权平均：原有库存成本 + 新入库成本）/ 总数量
    ▼
销售出库 → 销售成本 = 成品单位成本 × 出库数量
```

### 10.2 加权平均法

本项目推荐**移动加权平均法**，这是服装行业最常用的成本核算方式：

```
每次入库时重新计算单位成本：

新单位成本 = (原库存金额 + 本次入库金额) / (原库存数量 + 本次入库数量)

示例：
  原库存：100件 × 50元 = 5000元
  入库：  200件 × 55元 = 11000元
  
  新单位成本 = (5000 + 11000) / (100 + 200) = 53.33元
```

### 10.3 关键计算逻辑

**面料采购入库时：**

```
面料库存加权平均成本 = (原库存金额 + 本次入库金额) / (原库存数量 + 本次入库数量)

示例：
  原库存：500m × 45.00元 = 22,500元
  入库：  2400m × 48.00元 = 115,200元
  新成本 = (22,500 + 115,200) / (500 + 2400) = 47.50元/m
```

**生产领料出库时：**

```
领料成本 = 面料单位成本 × 领料数量

成本随出库从原料库存转到生产订单上：
  生产订单累计成本 += 领料成本
```

**成品生产入库时：**

```
成品单位成本 = 生产订单累计领料成本 / 完工数量

示例（黑色风衣 900件）：
  面料领料成本：1895.40m × 47.50 = 90,031.50元
  里布领料成本：1350.00m × 18.00 = 24,300.00元
  辅料领料成本：8 × 0.5 × 900 = 3,600.00元
  拉链领料成本：1 × 12.00 × 900 = 10,800.00元
  其他辅料：    3 × 0.5 × 900 = 1,350.00元
  包材：       2 × 2.50 × 900 = 4,500.00元
  ─────────────────────────────
  累计领料成本：134,581.50元
  
  成品单位成本 = 134,581.50 / 900 = 149.54元/件

成品入库加权平均：
  原库存：200件 × 145.00 = 29,000元
  入库：  900件 × 149.54 = 134,586元
  新成本 = (29,000 + 134,586) / (200 + 900) = 148.71元/件
```
