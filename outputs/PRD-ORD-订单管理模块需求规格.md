# PRD-ORD 订单管理模块需求规格

> 模块编号：ORD  
> 优先级：P0（第二期交付）  
> 前置依赖：基础数据模块（SPU/SKU、客户、季节、编码规则）

---

## 一、模块概述

订单管理模块是经纬系统的业务引擎，管理销售订单和生产订单的全生命周期。核心特色是服装行业特有的尺码矩阵交互和销售订单驱动生产订单的转化机制。

---

## 二、用户故事

### US-ORD-001 销售订单创建

**作为** 业务员  
**我希望** 以尺码矩阵的方式录入销售订单  
**以便** 高效地按款式和颜色录入各尺码的需求量

**验收标准：**

1. 创建订单时选择客户、季节、交货日期、业务员
2. 添加订单行时选择款式，系统自动加载该款式的颜色列表
3. 选择一个颜色后，系统渲染尺码矩阵（行为颜色，列为尺码），在单元格中填写数量
4. 支持批量填数：整行填相同数量、按比例分配（如2:3:3:2:1）
5. 每个颜色款一行数据，包含该颜色各尺码的数量
6. 自动计算每行小计和订单总数量
7. 输入单价后自动计算行金额 = 总数量 × 单价
8. 支持行折扣率输入，行实际金额 = 行金额 × 折扣率
9. 支持整单折扣金额输入，实际金额 = 订单总金额 - 整单折扣
10. 订单编号由编码规则自动生成

**边界场景：**

- 同一订单添加同一款式同一颜色 → 不允许，提示"该款式颜色已存在"
- 尺码数量输入0或负数 → 前端拦截
- 不填单价直接提交 → 允许（有些订单价格后续确认）
- 交货日期早于订单日期 → 前端警告但允许（可能是补录历史订单）

### US-ORD-002 销售订单状态流转

**作为** 业务员/业务经理  
**我希望** 销售订单按状态机流转  
**以便** 订单生命周期清晰可控

**验收标准：**

1. 订单状态包括：草稿、已确认、生产中、备货完成、已发货、已完成、已取消
2. 草稿→已确认：业务员提交，系统自动预留可用库存
3. 已确认→生产中：关联生产订单后自动流转
4. 已确认→已取消：仅未关联生产订单时可取消，同时释放预留库存
5. 生产中→备货完成：所有SKU库存满足时流转
6. 备货完成→已发货：出库单发货确认后自动流转
7. 已发货→已完成：确认签收
8. 每次状态变更记录操作人、时间、原因
9. 前端根据当前状态动态显示可操作按钮（不可操作的不显示）

---

### US-ORD-003 销售订单审批

**作为** 业务经理  
**我希望** 审批业务员提交的销售订单  
**以便** 控制订单质量和价格

**验收标准：**

1. 业务员提交订单后，订单进入"待审批"状态
2. 业务经理在审批列表中查看待审批订单
3. 审批通过 → 订单变为"已确认"，触发库存预留
4. 审批驳回 → 订单退回"草稿"，业务员可修改后重新提交
5. 审批时可填写审批意见

---

### US-ORD-004 销售订单数量变更

**作为** 业务员  
**我希望** 已确认的订单修改数量时走变更单审批  
**以便** 数量变更影响下游时可追溯和评估

**验收标准：**

1. 已确认状态的订单，直接编辑数量不可用
2. 发起数量变更时，展示当前矩阵，用户修改后系统计算差异
3. 差异矩阵展示每个SKU的增减量
4. 变更单需审批通过后才生效
5. 审批通过后，系统自动调整库存预留、通知生产/采购模块
6. 变更历史记录在订单跟踪时间线中

---

### US-ORD-005 订单转化（销售→生产）

**作为** 计划员  
**我希望** 从销售订单生成生产订单  
**以便** 根据销售需求安排生产

**验收标准：**

1. 选择已确认的销售订单行，点击"生成生产订单"
2. 系统自动检查库存满足情况：
   - 库存充足 → 不生成生产订单，直接锁定库存
   - 库存不足 → 缺口部分生成生产订单
3. 同款不同客户的销售需求可合并为一张生产订单
4. 一张销售订单可拆分为多张生产订单（不同款交期不同）
5. 生成生产订单时自动关联BOM（从SPU的默认BOM获取）
6. 销售订单与生产订单的多对多关系可查看

---

### US-ORD-006 生产订单管理

**作为** 计划员/计划经理  
**我希望** 管理生产订单的生命周期  
**以便** 跟踪每个款式在生产车间的进度

**验收标准：**

1. 生产订单可从销售订单转化，也可独立创建
2. 生产订单状态：草稿→已下达→已排产→裁剪中→缝制中→后整中→已完工→已入库
3. 每行有独立状态（同一订单不同颜色可能在不同生产阶段）
4. 主表状态取所有行的最滞后状态
5. 支持标记"跳过裁剪"（针织类产品），跳过后从"已排产"直接到"缝制中"
6. 每次状态变更记录在操作日志中
7. 生产进度页面展示各行的当前状态和完工数量

---

### US-ORD-007 订单跟踪时间线

**作为** 业务员/计划员  
**我希望** 查看订单的完整变更历史  
**以便** 了解订单从创建到完成的整个过程

**验收标准：**

1. 订单详情页展示时间线，按时间倒序排列
2. 时间线记录包含：操作时间、操作人、操作类型、变更内容
3. 状态变更、数量变更、交期变更、关联生产订单等均记录
4. 数量变更可展开查看差异矩阵

---

### US-ORD-008 库存满足率查看

**作为** 业务员  
**我希望** 在订单详情页看到每个SKU的库存满足情况  
**以便** 判断能否按时交货

**验收标准：**

1. 订单详情页展示库存满足矩阵，按颜色×尺码展示
2. 每个单元格显示：订单需求量 / 可用库存量 / 缺口量
3. 满足的标绿色，部分满足标黄色，不满足标红色
4. 可用库存为实时查询（非缓存）

---

### US-ORD-009 退货管理

**作为** 业务员/仓库主管  
**我希望** 管理客户退货、物流拒收和经销商退货  
**以便** 退回的成品经过质检后正确入库

> 详细设计方案见 08-设计补充文档 第三节「退货管理」，包含完整的状态流转、表结构和库存交互逻辑。

**验收标准：**

1. 可创建退货单，选择退货类型：客户退货（CUSTOMER_REJECT）、物流拒收（LOGISTICS_REJECT）、经销商退货（DISTRIBUTOR_RETURN）
2. 退货单关联原销售订单，按尺码矩阵录入退货数量
3. 退货申请需审批（走统一审批引擎）
4. 审批通过后生成退货入库单，仓库收货
5. 收货后进入质检，合格入可用库存，不合格标记报废/次品
6. 退货单状态：DRAFT → PENDING_APPROVAL → APPROVED → RECEIVING → QC → COMPLETED（审批驳回为 REJECTED）
7. 更新原销售订单的发货/退货状态

**边界场景：**

- 退货数量超过原订单对应SKU的发货数量 → 不允许，提示"退货数量不能超过发货数量"
- 退货时原订单已取消 → 允许退货（可能已发货后取消）

## 三、业务规则

| 编号 | 规则 |
|------|------|
| BR-ORD-001 | 草稿状态可自由编辑，确认后仅可改交期和备注 |
| BR-ORD-002 | 已确认订单改数量走变更单审批 |
| BR-ORD-003 | 销售订单确认时自动预留可用库存 |
| BR-ORD-004 | 销售订单取消时释放已预留库存 |
| BR-ORD-005 | 同款同色不可在同一订单中重复添加 |
| BR-ORD-006 | 生产订单行独立状态，主表取最滞后状态 |
| BR-ORD-007 | 生产订单必须关联BOM才能下达 |
| BR-ORD-008 | 跳过裁剪环节需标记，记录在变更日志 |
| BR-ORD-009 | 退货数量不能超过原订单对应SKU的发货数量 |
