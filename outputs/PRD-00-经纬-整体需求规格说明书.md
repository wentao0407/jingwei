## 一、项目概述

### 1.1 项目背景

本项目为一家服装生产销售企业量身打造全新管理信息系统，覆盖从客户下单到生产制造再到仓储物流的全链路业务。系统不与任何现有系统对接，为全新建设。

### 1.2 项目目标

- 建立统一的服装行业主数据标准（SPU/SKU、BOM、季节、尺码组）
- 实现销售订单驱动的生产排产与采购计算闭环
- 实现四类库存（可用/锁定/质检/在途）的精细化管理，杜绝超卖超发
- 实现仓库作业标准化（收货→质检→上架、波次→拣货→复核→发货）
- 所有业务数据变更可追溯、可审计
- 为未来多厂区扩展预留架构空间

### 1.3 项目范围

**包含：**

| 业务域 | 覆盖范围 |
|--------|---------|
| 基础数据 | 物料主数据、SPU/SKU、季节、尺码组、编码规则、供应商、客户、仓库/库位 |
| 订单管理 | 销售订单、生产订单、订单转化、变更管理 |
| 采购管理 | BOM管理、MRP计算、采购订单、到货检验、供应商对账 |
| 库存管理 | 四类库存、库存预留/释放、出入库单据、调拨、盘点、预警 |

### 1.4 术语定义

| 术语 | 全称 | 定义 |
|------|------|------|
| SPU | Standard Product Unit | 标准产品单元，即"款式" |
| SKU | Stock Keeping Unit | 最小库存单元，即"颜色+尺码"的组合 |
| Color-way | 颜色款 | SPU下的颜色维度，介于SPU和SKU之间。一个颜色款对应一种颜色，其下按尺码展开为多个SKU |
| BOM | Bill of Materials | 物料清单，定义成品与面料/辅料/包材的用量关系 |
| MRP | Material Requirements Planning | 物料需求计划，根据生产订单+BOM计算采购需求 |
| ASN | Advanced Shipping Notice | 到货通知单，供应商发货前通知仓库准备收货 |
| FIFO | First In First Out | 先进先出，出库时优先出早期入库的批次 |
| 可用库存 | Available | 可自由分配给新订单的库存 |
| 锁定库存 | Locked | 已分配给销售订单但尚未出库的库存 |
| 质检库存 | QC | 已到货但尚未完成检验的库存 |
| 在途库存 | In-Transit | 已下采购订单但尚未到货的数量 |
| MOQ | Minimum Order Quantity | 最小起订量，供应商的最低采购数量 |
| QC | Quality Control | 质量控制/来料检验 |
| 波次 | Wave | 将多个出库需求合并为一批，一起拣货，提高效率 |
| 盲盘 | Blind Count | 盘点时不展示系统数量，只填写实盘数量 |

## 三、业务流程总览

### 3.1 核心业务主流程

```
客户下单 → 销售订单 → 排产 → 生产订单 → MRP计算 → 采购 → 到货 → 检验 → 入库
    │                                              │
    │                                              └→ 领料出库 → 生产 → 完工入库
    │                                                            │
    └←─────────────────────────── 备货 → 拣货 → 发货 ←───────────┘
```

### 3.2 业务流程详细说明

**流程一：以销定产（主流程）**

1. 业务员创建销售订单，按尺码矩阵录入各颜色各尺码的需求量
2. 业务经理审批销售订单
3. 订单确认后，系统自动预留可用库存；不足部分记录缺口
4. 计划员从销售订单生成生产订单
5. 系统根据生产订单+BOM执行MRP计算，生成采购建议
6. 采购员审核采购建议，生成采购订单
7. 采购经理审批采购订单，下发给供应商
8. 供应商发货 → 仓库收货 → 质检 → 入库
9. 车间领料 → 生产 → 完工入库
10. 备货完成 → 波次拣货 → 复核打包 → 发货

**流程二：备货生产**

1. 计划员根据预测独立创建生产订单
2. 后续步骤同流程一的步骤5-10

**流程三：库存盘点**

1. 仓库主管创建盘点单
2. 仓管员逐项实盘（盲盘模式）
3. 系统自动比对差异
4. 仓库主管审核差异，安排复盘
5. 确认差异后系统自动调整库存

| 恢复 | RPO ≤ 1小时，RTO ≤ 4小时 |

## 六、约束与假设

### 6.1 技术约束

| 约束 | 说明 |
|------|------|
| 后端框架 | Java 17 + Spring Boot 3.x |
| 数据库 | PostgreSQL 15+ |
| 缓存 | Redis 7+（Session存储 + 幂等令牌 + 库存查询缓存 + 编码流水号备用） |
| 消息队列 | 初期不需要（Spring Event + Outbox），后期RabbitMQ |
| 前端框架 | React 18 + Ant Design Pro |
| 部署方式 | 初期JAR包 + systemd单机部署，后期容器化 |
| 文件存储 | 本地文件系统（文件量小，无需对象存储） |
| 库存扣减 | 数据库乐观锁为主（<20人并发，冲突率极低），热点SKU可降级Redis原子操作 |

**部署架构（基于<20人并发的简化方案）：**

```
Nginx（反向代理 + 静态资源）
    │
    ▼
Spring Boot JAR（单体应用）
    │
    ├── PostgreSQL（主库）
    └── Redis（Session + 缓存 + 幂等令牌）
```

**简化说明：** 基于<20人并发的小型规模，以下组件从初始方案中移除或简化：不引入 Spring Cloud Gateway（Nginx 直连即可）；不引入 Redis 库存扣减（乐观锁足够）；不引入 RabbitMQ（Outbox 足够）；不引入 MinIO（本地文件系统足够）；不引入 K8s（JAR包直跑即可）。

### 6.2 业务假设

| 假设 | 说明 |
|------|------|
| 单厂区启动 | 初期只有一个生产厂区，但架构支持多厂区 |
| 不对接外部系统 | 本期无ERP/财务/电商对接需求 |
| 不含MES | 车间设备级管控不在本期范围 |
| 扫码设备 | 使用USB扫码枪或安卓PDA，不采购专用设备 |
| 打印设备 | 使用普通A4激光打印机+标签打印机 |
