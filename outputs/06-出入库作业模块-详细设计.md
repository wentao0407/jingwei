# 出入库作业模块（Warehouse Operation）详细设计

> 版本：v1.0  
> 日期：2026-04-30  
> 适用行业：服装生产销售  
> 前置依赖：基础数据模块（仓库/库位、SKU）、库存管理模块（库存操作、库位推荐）、采购管理、订单管理

---

## 一、模块总览

前面几个模块解决的是"数据怎么存、逻辑怎么算"的问题。这个模块解决的是**仓库里的人怎么干活**的问题。

库存管理定义了"入库要扣什么字段、出库要走什么流水"，而出入库作业模块定义的是：

- 仓库人员拿到一张入库单，**先做什么、后做什么、每一步扫码还是手填**
- 50个销售订单要出库，是50个人各跑一趟，还是合并成一波高效拣货
- 拣完的货怎么复核、怎么打包、怎么交接给物流
- 所有单据怎么打印、条码怎么生成、扫码枪怎么对接

服装行业仓库作业的特殊性：

1. **成品出库以款式为单位** — 拣货时不是按SKU拣，而是按款式拣（黑色风衣S/M/L/XL一起拿），然后按订单分配
2. **面料入库需要量布** — 面料按米入库，需要核对实际米数和采购数量
3. **季节性爆仓** — 换季时出入库量可能是平时的5-10倍，波次管理必须高效
4. **成衣挂装出货** — 很多高档服装不是折叠装箱，而是衣架挂装，物流方式不同

本模块包含：
1. 入库作业 — 收货、质检、上架
2. 出库作业 — 波次生成、拣货、复核、发货
3. 波次管理 — 拣货策略与合并规则
4. 条码系统 — 编码、打印、扫码对接
5. 打印模板 — 各类单据的打印方案

## 二、入库作业

### 2.1 入库作业全流程

```
收到到货通知（ASN）
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 第1步：收货                                          │
│                                                       │
│  扫码/手工输入ASN单号 → 调出到货通知单信息             │
│  逐行核对物料 → 扫物料条码确认 → 填写实收数量          │
│  面料：核对卷数、目测门幅                              │
│  辅料：点数或称重核对                                  │
│  包材：清点数量                                       │
│                                                       │
│  可以部分收货（一次到一部分）                           │
│  收货后：在途库存 -= 实收数量                          │
│           质检库存 += 实收数量                          │
└────────────────────────────┬────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────┐
│ 第2步：来料检验（QC）                                 │
│                                                       │
│  面料检验项：                                         │
│    □ 色差（对比确认样/色卡）→ ≥4级合格                 │
│    □ 克重 → 标准值±10%以内合格                        │
│    □ 门幅 → 标准值±2cm以内合格                        │
│    □ 缩水率 → ≤3%合格                                │
│    □ 色牢度 → ≥3-4级合格                              │
│    □ 外观疵点 → 每100米≤3处合格                       │
│                                                       │
│  辅料检验项：                                         │
│    □ 规格/尺寸 → 对照BOM要求                          │
│    □ 颜色 → 对比确认样                                │
│    □ 数量 → 清点或称重                                │
│    □ 外观 → 无毛刺/无生锈/无脱色                      │
│                                                       │
│  检验结果：PASS / FAIL / CONCESSION（让步接收）        │
│                                                       │
│  PASS：  质检库存 -= 合格数量，可用库存 += 合格数量     │
│  FAIL：  质检库存 -= 不合格数量 → 生成退货出库单       │
│  CONCESSION：标记降级，可用库存 += 让步数量             │
└────────────────────────────┬────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────┐
│ 第3步：上架                                           │
│                                                       │
│  系统推荐库位（复用库存模块的库位推荐逻辑）             │
│  仓库人员确认或手动指定库位                            │
│  扫库位条码 → 扫物料条码 → 确认上架                    │
│                                                       │
│  更新库存记录的 location_id                            │
│  更新库位的占用状态                                    │
└─────────────────────────────────────────────────────┘
```

**PDA/扫码枪端的操作流程：**

```
1. 扫ASN单号条码 → 加载到货通知
2. 逐行显示待收货物料：
   ┌─────────────────────────────────┐
   │ 全棉斜纹布  FAB001              │
   │ 预计：2400m  已收：0m           │
   │                                 │
   │ [扫物料条码]                    │
   │                                │
   │ 实收数量：[____] m              │
   │ 实收卷数：[____] 卷             │
   │                                │
   │ [确认收货]  [跳过]  [异常上报]   │
   └─────────────────────────────────┘
3. 确认后 → 自动生成批次号 → 触发库存变更（在途→质检）
4. 全部收完 → 收货单状态变为COMPLETED
```

**部分收货处理：**

供应商一次送了一部分货，另一部分后续再送：

```
ASN预计：全棉斜纹布 2400m
第1次收货：1500m → 生成批次号 FAB001-20260515-001
第2次收货：900m  → 生成批次号 FAB001-20260518-002（不同日期不同批次）

关键：不同次到货必须分不同批次号
      因为可能不是同一天染的色，存在色差风险
```

### 2.4 上架操作

```java
/**
 * 上架服务
 */
@Service
public class PutawayService {

    /**
     * 推荐上架库位
     */
    public List<PutawaySuggestion> suggestLocations(Long receivingLineId) {
        WarehouseReceivingLine line = receivingLineRepo.findLineById(receivingLineId);
        
        List<PutawaySuggestion> suggestions = new ArrayList<>();
        
        // 规则1：同物料已有库存的库位（合并存放）
        List<Location> sameMaterialLocations = locationRepo.findLocationsWithMaterial(
            line.getMaterialId(), line.getWarehouseId());
        for (Location loc : sameMaterialLocations) {
            BigDecimal remainingCapacity = loc.getCapacity().subtract(loc.getUsedCapacity());
            if (remainingCapacity.compareTo(BigDecimal.ZERO) > 0) {
                suggestions.add(new PutawaySuggestion(loc, remainingCapacity, "SAME_MATERIAL"));
            }
        }
        
        // 规则2：同类物料的空闲存储位
        // 规则3：同区域的空闲存储位
        // ...
        
        // 按优先级排序
        suggestions.sort(Comparator.comparing(PutawaySuggestion::getPriority));
        return suggestions;
    }

    /**
     * 确认上架
     */
    @Transactional
    public void confirmPutaway(PutawayConfirmRequest request) {
        // 1. 更新收货行的上架状态和库位
        receivingLineRepo.updatePutaway(request.getReceivingLineId(), 
            request.getLocationId());
        
        // 2. 更新库存记录的库位信息
        inventoryRepo.updateLocation(request.getInventoryId(), request.getLocationId());
        
        // 3. 更新库位占用状态
        locationRepo.updateUsedCapacity(request.getLocationId(), request.getQuantity());
    }
}
```

### 3.2 出库单的详细状态机

```
DRAFT ──────→ CONFIRMED ──────→ PICKING ──────→ SORTING ──────→ CHECKING ──────→ PACKING ──────→ SHIPPED
  │              │                 │                │                │               │
  │              │                 │                │                │               │
  └→ CANCELLED   │                 │                │                │               │
                 │                 │                │                │               │
                 └→ CANCELLED      │                │                │               │
                                   │                │                │               │
                                   └→ 缺货退回       └→ 分拣异常      └→ 复核不通过   └→ CANCELLED
                                      CONFIRMED        SORTING         PICKING
```

**SORTING（分拣）环节可选：** 逐单拣货时不需要分拣，直接从 PICKING → CHECKING。只有汇总拣货时才需要分拣步骤。

### 3.4 拣货单行的排序策略

拣货单行的顺序直接影响拣货效率。仓库人员不应该在仓库里来回跑。

**路径优化策略：**

```
仓库布局示意：
  ┌──────┬──────┬──────┬──────┐
  │ A-01 │ A-02 │ A-03 │ A-04 │  ← 库区A（存储位）
  ├──────┼──────┼──────┼──────┤
  │ B-01 │ B-02 │ B-03 │ B-04 │  ← 库区B（存储位）
  ├──────┼──────┼──────┼──────┤
  │ C-01 │ C-02 │ C-03 │ C-04 │  ← 库区C（拣货位）
  └──────┴──────┴──────┴──────┘
         ↑ 入口

拣货路线优化（S型走位）：
  入口 → C-01 → C-02 → C-03 → C-04 → B-04 → B-03 → B-02 → B-01 → A-01 → ... → 出口
```

**排序规则：**

1. 拣货位优先于存储位（减少进入深处库区）
2. 同库区内按库位编码顺序（S型走位）
3. 同库位多SKU按编码顺序
4. 服装行业特殊：同款式同颜色的不同尺码排在一起（方便一次拿齐）

```java
public List<PickOrderLine> optimizeRoute(List<PickOrderLine> lines) {
    return lines.stream()
        .sorted(Comparator
            // 先按库位类型：PICKING < STORAGE
            .comparing(line -> getLocationTypePriority(line.getLocationId()))
            // 再按库区编码
            .thenComparing(line -> getZoneCode(line.getLocationId()))
            // 再按库位排序号（S型走位）
            .thenComparing(line -> getLocationSortOrder(line.getLocationId()))
            // 最后按款式+颜色分组，尺码顺序
            .thenComparing(line -> line.getSpuId())
            .thenComparing(line -> line.getColorWayId())
            .thenComparing(line -> line.getSizeSortOrder())
        )
        .collect(Collectors.toList());
}
```

## 四、波次管理

### 4.1 什么是波次？

波次（Wave）是把多个出库需求合并成一批，一起拣货，提高效率。

```
没有波次：
  50个销售订单 → 50次拣货 → 50次走仓库 → 效率极低

有波次：
  50个销售订单 → 合并为3个波次 → 3次拣货 → 效率提升10倍以上
```

### 4.2 波次生成策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 按时间段 | 每小时生成一个波次，收集该时段所有待出库订单 | 日常出库 |
| 按订单量 | 待出库订单达到N个时生成一个波次 | 中等单量 |
| 按物流商 | 同一物流公司的订单合并一个波次 | 物流分拨 |
| 按目的地 | 同城市/同区域的订单合并 | 区域配送 |
| 手动创建 | 手动选择订单创建波次 | 特殊场景 |

**服装行业推荐：按时间段 + 按物流商** 组合策略。

### 4.4 波次释放与拣货单生成

**波次创建后，需要"释放"才能生成拣货单。** 释放过程：

```
波次（CREATED）
    │
    ▼ 释放
    │
    ├── 1. 校验：所有关联出库单的库存是否都满足
    │       不满足 → 标记缺货行 → 可以选择部分释放或等待补货
    │
    ├── 2. 分配库存：为每个出库单行指定具体库位和批次
    │       按FIFO + 批次优先原则
    │       面料/成品同款同色优先从同一批次分配
    │
    ├── 3. 生成拣货单：
    │       逐单拣模式 → 每个出库单一张拣货单
    │       汇总拣模式 → 一个波次一张拣货单（所有出库单汇总）
    │
    ├── 4. 拣货单行排序：按路径优化策略排序
    │
    └── 5. 波次状态 → RELEASED
```

### 4.5 汇总拣货 vs 逐单拣货

```
场景：3个销售订单，都包含黑色M码风衣

逐单拣货（按订单）：
  拣货单1（订单A）：去C-01拿黑色M码10件
  拣货单2（订单B）：去C-01拿黑色M码5件
  拣货单3（订单C）：去C-01拿黑色M码8件
  → 跑3趟同一个库位

汇总拣货（按SKU）：
  拣货单：去C-01拿黑色M码23件
  → 跑1趟，到分拣区再按订单分配
  → 效率提升3倍
```

**汇总拣货需要分拣步骤：**

```
汇总拣完23件黑色M码风衣
    │
    ▼ 分拣
  ┌──────────┬──────────┬──────────┐
  │ 订单A     │ 订单B     │ 订单C     │
  │ 10件     │ 5件      │ 8件      │
  └──────────┴──────────┴──────────┘
  逐件扫码 → 系统自动识别该分配给哪个订单
```

**选择策略的判断逻辑：**

```java
public PickType determinePickType(WarehouseWave wave) {
    int orderCount = wave.getOrderCount();
    int totalLines = wave.getTotalLines();
    
    // 订单数少、行数多 → 逐单拣（每个订单内容丰富，跑一趟值得）
    if (orderCount <= 5 && totalLines > 20) {
        return PickType.ORDER;
    }
    
    // 订单数多、行数少 → 汇总拣（大量小单，合并更高效）
    if (orderCount > 10 && totalLines < orderCount * 3) {
        return PickType.STYLE;
    }
    
    // 默认汇总拣
    return PickType.STYLE;
}
```

### 4.6 领料出库（生产场景）

除了销售出库，还有生产领料出库 — 从原料仓领面料和辅料去生产：

```
生产订单（已排产）
    │
    ▼
领料出库流程
    │
    ├── 1. 根据BOM + 生产数量计算各原料需求
    ├── 2. 检查预留状态（生产订单下达时已预留）
    ├── 3. 生成领料出库单
    │       面料：按款式+颜色分行，每行指定批次
    │       辅料：按物料分行
    ├── 4. 生成领料拣货单
    │       面料通常在存储位 → 推荐路径
    │       辅料通常在拣货位 → 推荐路径
    ├── 5. 拣货 → 复核 → 交接给车间
    └── 6. 确认出库 → 扣减库存
```

**面料领料的特殊性：**

面料是整卷的，一卷60-100米。如果需要23米，有两种处理方式：

- **整卷领出** — 把整卷60米都领到车间，用完退回剩余的37米（退料入库）
- **裁剪领出** — 在仓库把23米从卷上剪下来，只领23米，剩余留在仓库

**系统支持两种模式，由仓库管理员在领料时选择。** 整卷领出时，出库数量填整卷米数；退料时走退料入库流程。

        if (barcodeBuffer.length > 3) {
            // 条码扫描完成，触发业务逻辑

    } else if (barcode.startsWith('BX')) {
        handleCaseScan(barcode);
    } else {
        // 尝试作为通用单据号查询
        handleDocumentScan(barcode);

### 6.4 HTML模板方案细节

```java
@Service
public class PrintService {

    /**
     * 渲染打印模板
     */
    public byte[] renderPdf(String templateCode, Map<String, Object> data) {
        // 1. 加载模板
        PrintTemplate template = templateRepo.findByCode(templateCode);
        
        // 2. 填充数据（使用 Thymeleaf 模板引擎）
        String html = templateEngine.process(template.getTemplateContent(), data);
        
        // 3. HTML → PDF（使用 OpenHTMLToPDF 或 Flying Saucer）
        try (ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            PdfRendererBuilder builder = new PdfRendererBuilder();
            builder.withHtmlContent(html, null);
            builder.toStream(baos);
            builder.run();
            return baos.toByteArray();
        }
    }
}
```

## 八、领域事件

出入库作业模块发布的事件：

| 事件名 | 触发时机 | 订阅方 |
|--------|---------|--------|
| WarehouseGoodsReceivedEvent | 收货确认 | 库存模块（在途→质检库存） |
| WarehouseQcPassedEvent | 质检合格 | 库存模块（质检库存→可用库存），采购模块（回写asn_line.qc_status） |
| WarehouseQcFailedEvent | 质检不合格 | 采购模块（触发退货） |
| PickCompletedEvent | 拣货完成 | 通知模块（通知复核人员） |
| OutboundShippedEvent | 发货确认 | 库存模块（扣减锁定库存）、订单模块（更新状态） |
| MaterialReturnedEvent | 退料入库 | 库存模块（增加可用库存）、生产模块（更新领料记录） |

> **关于来料检验（QC）的职责边界：** 采购模块负责定义检验标准（检验项、合格标准），出入库模块负责执行检验操作和记录检验结果。检验完成后，出入库模块发布 `WarehouseQcPassedEvent` / `WarehouseQcFailedEvent`，采购模块订阅并回写 `procurement_asn_line.qc_status` 和 `qc_result`，避免两处独立维护。
