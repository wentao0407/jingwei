# 采购管理模块（Procurement Management）详细设计

> 版本：v1.0  
> 日期：2026-04-30  
> 适用行业：服装生产销售  
> 前置依赖：基础数据模块（物料、供应商、编码规则）、订单管理模块（生产订单、BOM）

---

## 一、模块总览

采购管理模块解决的核心问题：**生产订单创建后，需要买多少面料、多少辅料、找谁买、什么时候买、买回来质量合不合格。**

服装行业采购的特殊性：

1. **面料用量跟尺码相关** — L码比S码多用15-20%的面料，不能按"平均用量×总件数"粗算
2. **面料有损耗率** — 排料裁剪会产生损耗（5%-15%不等），不同面料损耗不同
3. **同款同色必须同批面料** — 不同批次的色差在服装行业是不可接受的
4. **面料采购周期长** — 印花/染色面料30-60天，辅料7-15天，必须提前算准
5. **供应商有起订量** — 面料一缸起订几百米，纽扣一袋起订几千颗

本模块包含：
1. BOM管理 — 物料清单的维护与版本控制
2. MRP计算引擎 — 需求展开、库存抵扣、采购建议
3. 采购订单管理 — 下单、供应商分配、跟踪
4. 采购到货与来料检验
5. 供应商对账

## 二、BOM管理

### 2.1 服装BOM的特点

与机械制造BOM的关键区别：

| 维度 | 机械BOM | 服装BOM |
|------|---------|---------|
| 层级 | 多层级（组件→部件→零件） | 通常单层级（成品→面料/辅料） |
| 用量单位 | 固定（1个零件/1台） | 面料按米/码，辅料按个/套 |
| 尺码关联 | 无 | 面料用量随尺码变化 |
| 损耗 | 固定废品率 | 面料有排料损耗率，不同面料不同 |
| 批次要求 | 通常无 | 同款同色必须同批次面料 |

### 2.3 尺码用量表结构（size_consumptions）

`SIZE_DEPENDENT` 类型的BOM行，用 JSONB 存储每个尺码的用量系数：

```json
{
  "baseSizeId": 11,
  "baseSizeCode": "M",
  "baseConsumption": 1.80,
  "sizes": [
    {"sizeId": 10, "code": "S",   "consumption": 1.60},
    {"sizeId": 11, "code": "M",   "consumption": 1.80},
    {"sizeId": 12, "code": "L",   "consumption": 1.95},
    {"sizeId": 13, "code": "XL",  "consumption": 2.10},
    {"sizeId": 14, "code": "XXL", "consumption": 2.30}
  ]
}
```

**说明：** 
- `baseConsumption` 和 `baseSizeId` 与主表的 `base_consumption` / `base_size_id` 一致，是冗余存储，方便JSONB独立使用
- 每个尺码的 `consumption` 是**绝对用量**（米/件），不是系数。直接用更直观，避免"系数×基准"的计算歧义
- 用量数据通常由技术/版房提供，不是系统推算的

### 2.5 BOM版本控制

**数据库级唯一约束：**

```sql
-- 保障"同一SPU仅一个APPROVED版本"的业务规则
CREATE UNIQUE INDEX uk_bom_spu_approved ON bom (spu_id) 
    WHERE status = 'APPROVED' AND deleted = false;
```

BOM必须支持版本控制，原因：
- 生产过程中可能换面料（原面料停产/色差问题）
- 同一款不同季节的BOM可能微调
- 已在生产的订单用旧版BOM，新订单用新版BOM

**规则：**
- 同一 SPU 只能有一个 APPROVED 状态的 BOM
- 新建 BOM 版本时，旧版本自动设为 OBSOLETE（如果已有新版本 APPROVED）
- 生产订单关联 BOM 时，记录具体的 bom_id，不受后续版本变更影响

```
SPU20260001 的 BOM 版本历史：

v1  APPROVED   2025-12-01 ~ 2026-02-15  旧版（已淘汰）
v2  APPROVED   2026-02-16 ~ NULL         当前版本（面料换了供应商）
v3  DRAFT      未生效                      新版开发中
```

## 三、MRP计算引擎

### 3.1 MRP的核心逻辑

MRP（Material Requirements Planning）要回答三个问题：
1. **需要多少？** — 根据生产订单 + BOM 展开
2. **已有多少？** — 当前库存 - 已预留库存 + 在途数量
3. **还要买多少？** = 需要量 - 已有量，再考虑起订量和采购倍数

### 3.2 计算流程

```
┌──────────────────────────────────────────────────────────────┐
│                      MRP 计算流程                             │
│                                                              │
│  1. 需求汇总                                                  │
│  ┌──────────────┐                                            │
│  │ 生产订单      │ ──→ 按 SPU+Color-way 汇总各尺码生产数量      │
│  └──────────────┘                                            │
│         │                                                    │
│         ▼                                                    │
│  2. BOM 展开                                                  │
│  ┌──────────────┐                                            │
│  │ BOM          │ ──→ 每个物料 × 各尺码数量 = 物料总需求量       │
│  └──────────────┘     （面料：尺码用量×尺码件数×(1+损耗率)）     │
│         │              （辅料：单件用量×总件数）                  │
│         ▼                                                    │
│  3. 同物料合并                                                │
│  不同款式用的同种面料 → 合并需求（一起买可以拿更好价格）          │
│         │                                                    │
│         ▼                                                    │
│  4. 库存抵扣                                                  │
│  ┌──────────────┐                                            │
│  │ 当前库存      │ ──→ 可用库存 = 实际库存 - 锁定库存            │
│  │ 在途数量      │ ──→ 加上已下采购单但未到货的数量               │
│  └──────────────┘     净需求 = 总需求 - 可用库存 - 在途数量      │
│         │                                                    │
│         ▼                                                    │
│  5. 采购建议                                                  │
│  净需求 → 考虑起订量(MOQ) → 考虑采购倍数 → 生成采购建议行       │
│         │                                                    │
│         ▼                                                    │
│  6. 供应商匹配                                                │
│  为每个采购建议推荐最优供应商                                   │
│  （基于：供应关系、历史价格、交期、优先级）                      │
└──────────────────────────────────────────────────────────────┘
```

### 3.3 详细计算过程

#### 第一步：需求汇总

输入：一张或多张生产订单

```
生产订单 PO-20260430-001：

款式：2026春季女士风衣（SP20260001）
BOM版本：v2

黑色款：
  S: 100件, M: 200件, L: 300件, XL: 200件, XXL: 100件
  小计：900件

卡其色款：
  S: 50件, M: 100件, L: 150件, XL: 100件, XXL: 50件
  小计：450件
```

#### 第二步：BOM展开

以面料"全棉斜纹布"为例（SIZE_DEPENDENT，损耗率8%）：

**黑色款的面料需求：**

```
S码:   1.60m × 100件 × (1 + 0.08) = 172.80m
M码:   1.80m × 200件 × (1 + 0.08) = 388.80m
L码:   1.95m × 300件 × (1 + 0.08) = 631.80m
XL码:  2.10m × 200件 × (1 + 0.08) = 453.60m
XXL码: 2.30m × 100件 × (1 + 0.08) = 248.40m
                                    ──────────
黑色款面料小计：                      1895.40m
```

**卡其色款的面料需求：**

```
S码:   1.60m ×  50件 × 1.08 = 86.40m
M码:   1.80m × 100件 × 1.08 = 194.40m
L码:   1.95m × 150件 × 1.08 = 315.90m
XL码:  2.10m × 100件 × 1.08 = 226.80m
XXL码: 2.30m ×  50件 × 1.08 = 124.20m
                               ──────────
卡其色款面料小计：               947.70m
```

**合计：全棉斜纹布需求 = 1895.40 + 947.70 = 2843.10m**

以辅料"4孔树脂扣"为例（FIXED_PER_PIECE，8颗/件，损耗0%）：

```
总件数 = 900 + 450 = 1350件
需求量 = 8 × 1350 = 10800颗
```

#### 第三步：同物料合并

如果另一张生产订单也用了"全棉斜纹布"（不同款式但同面料），需求量合并：

```
PO-20260430-001（风衣）：全棉斜纹布 2843.10m
PO-20260430-002（夹克）：全棉斜纹布 560.00m
─────────────────────────────────────────
合并需求：                全棉斜纹布 3403.10m
```

#### 第四步：库存抵扣

```
全棉斜纹布的库存情况：
  实际库存：800m
  锁定库存：200m（已分配给其他生产订单）
  在途数量：500m（已下采购单，未到货）
  
可用库存 = 800 - 200 = 600m
在途数量 = 500m

净需求 = 3403.10 - 600 - 500 = 2303.10m
```

#### 第五步：采购建议

```
净需求：2303.10m

供应商起订量(MOQ)：500m
采购倍数：100m（必须整百米采购）

计算：
  2303.10 → 向上取整到100的倍数 → 2400m
  2400 ≥ 500（满足起订量）
  
建议采购量：2400m
```

### 3.5 MRP计算引擎的代码设计

```java
/**
 * MRP计算引擎 — 无状态服务，可单例
 */
@Service
public class MrpEngine {

    private final BomRepository bomRepository;
    private final InventoryQueryService inventoryQueryService;
    private final SupplierRecommendationService supplierRecommendationService;
    private final ProcurementOrderRepository procurementOrderRepository;

    /**
     * 执行MRP计算
     * @param request MRP计算请求（包含选中的生产订单行）
     * @return MRP计算结果
     */
    public MrpCalculationResult calculate(MrpRequest request) {
        String batchNo = codingRuleService.generateCode("MRP_BATCH", Map.of());

        
        // ========== 第1步：需求汇总 ==========
        // 按生产订单行收集各 SPU+Color-way 的尺码数量
        List<DemandItem> demandItems = collectDemands(request.getProductionLineIds());

        
        // ========== 第2步：BOM展开 ==========
        // 对每个需求项，展开BOM，计算每个物料的用量
        List<MaterialDemand> materialDemands = expandBom(demandItems);

        
        // ========== 第3步：同物料合并 ==========
        // 不同款式/颜色使用的同种物料，需求量合并
        Map<Long, MaterialDemand> mergedDemands = mergeByMaterial(materialDemands);

        
        // ========== 第4步：库存抵扣 ==========
        // 查询可用库存和在途数量，计算净需求
        List<MrpResultItem> results = deductInventory(mergedDemands);

        
        // ========== 第5步：采购建议 ==========
        // 考虑起订量、采购倍数，生成建议采购量
        applyPurchaseConstraints(results);

        
        // ========== 第6步：供应商匹配 ==========
        // 为每个采购建议推荐供应商
        matchSuppliers(results);

        
        // ========== 组装结果 ==========
        MrpCalculationResult result = new MrpCalculationResult();
        result.setBatchNo(batchNo);
        result.setItems(results);
        result.setSources(buildSources(demandItems, batchNo));
        return result;
    }


    /**
     * BOM展开 — 核心计算逻辑
     */
    private List<MaterialDemand> expandBom(List<DemandItem> demandItems) {
        List<MaterialDemand> materialDemands = new ArrayList<>();

        for (DemandItem demand : demandItems) {
            // 获取当前生效的BOM
            Bom bom = bomRepository.findActiveBySpuId(demand.getSpuId())
                .orElseThrow(() -> new BizException(
                    "款式 %s 没有生效的BOM", demand.getSpuCode()));

            for (BomItem bomItem : bom.getItems()) {
                BigDecimal quantity = calculateMaterialQuantity(bomItem, demand);
                
                materialDemands.add(new MaterialDemand(
                    demand.getProductionLineId(),
                    demand.getSpuId(),
                    demand.getColorWayId(),
                    bomItem.getMaterialId(),
                    bomItem.getMaterialType(),
                    quantity,
                    bom.getId(),
                    buildDetail(bomItem, demand)  // 记录各尺码的计算过程
                ));
            }
        }
        return materialDemands;
    }

    /**
     * 计算单个物料在某个颜色款上的总用量
     */
    private BigDecimal calculateMaterialQuantity(BomItem bomItem, DemandItem demand) {
        switch (bomItem.getConsumptionType()) {
            
            case FIXED_PER_PIECE: {
                // 每件固定用量 × 总件数
                int totalPieces = demand.getSizeMatrix().getTotalQuantity();
                return bomItem.getBaseConsumption()
                    .multiply(BigDecimal.valueOf(totalPieces));
            }
            
            case SIZE_DEPENDENT: {
                // 各尺码用量 × 对应件数，求和，再乘(1+损耗率)
                BigDecimal total = BigDecimal.ZERO;
                SizeConsumptionMap sizeMap = bomItem.getSizeConsumptions();
                
                for (SizeQuantity sq : demand.getSizeMatrix().getSizes()) {
                    BigDecimal consumption = sizeMap.getConsumption(sq.getSizeId());
                    BigDecimal subtotal = consumption
                        .multiply(BigDecimal.valueOf(sq.getQuantity()));
                    total = total.add(subtotal);
                }
                
                // 加损耗
                BigDecimal wastageMultiplier = BigDecimal.ONE
                    .add(bomItem.getWastageRate());
                return total.multiply(wastageMultiplier)
                    .setScale(2, RoundingMode.HALF_UP);
            }
            
            case PER_ORDER: {
                // 按订单整体用量（如唛头，一个款1套，不管多少件）
                return bomItem.getBaseConsumption();
            }
            
            default:
                throw new BizException("未知的消耗类型: " + bomItem.getConsumptionType());
        }
    }
}
```

### 3.6 MRP计算的触发时机

| 触发方式 | 说明 | 场景 |
|---------|------|------|
| 手动触发 | 用户选择生产订单行，点击"计算采购需求" | 正常排产流程 |
| 事件触发 | 订阅 `ProductionOrderCreatedEvent` | 自动计算（可配置开关） |
| 定时触发 | 每日凌晨扫描未计算的生产订单 | 兜底机制 |
| 变更触发 | 订单数量变更审批通过后 | 重新计算增量需求 |

**事件触发流程：**

```
生产订单创建
    │
    ▼
ProductionOrderCreatedEvent
    │
    ▼
MrpEventSubscriber.onProductionOrderCreated()
    │
    ├── 检查：该生产订单是否有BOM？有 → 继续
    ├── 检查：是否开启自动MRP？是 → 继续
    ├── 执行MRP计算
    └── 生成采购建议（状态：PENDING）
```

## 四、采购订单管理

### 4.1 状态机

```
                    ┌──────────┐
```

### 4.3 从MRP结果生成采购订单

**操作流程：**

1. 用户查看MRP计算结果列表
2. 勾选需要采购的行（可按供应商分组批量创建）
3. 点击"生成采购订单"
4. 系统按供应商分组，每个供应商生成一张采购订单
5. 自动填充：物料、数量、建议供应商、最近采购价
6. 用户可调整数量、单价、交期
7. 提交后，MRP结果行状态变为 CONVERTED

**同一供应商不同物料的合并：**

```
MRP结果（按供应商分组前）：
  全棉斜纹布  2400m  → 供应商A
  涤纶里布    1500m  → 供应商A
  4孔树脂扣  10800颗 → 供应商B
  YKK拉链    1350条  → 供应商C

按供应商分组后：
  采购订单1（供应商A）：全棉斜纹布 2400m + 涤纶里布 1500m
  采购订单2（供应商B）：4孔树脂扣 10800颗
  采购订单3（供应商C）：YKK拉链 1350条
```

### 4.4 同款同色同批面料约束

这是一个关键的服装行业业务规则：**同一个款同一个颜色，必须使用同一批次的面料。**

不同批次面料存在色差，如果黑色风衣用了两批面料，做出来的衣服色差肉眼可见，客户必然退货。

**系统保障：**
- 采购时，同一 SPU + Color-way 的面料需求必须合并在同一张采购订单的同一行
- 不允许拆分到不同采购订单（系统校验）
- 到货入库时，同一采购订单行的面料必须入同一批次（系统自动分配批次号）
- 生产领料出库时，同一生产订单行的面料必须从同一批次出（系统推荐库位时校验）

## 五、采购到货与来料检验

> **职责边界说明：** 采购模块负责定义检验标准（哪些物料需要检验、检验项是什么、合格标准是什么），出入库模块负责执行检验操作和记录检验结果。`procurement_asn_line.qc_status` 和 `qc_result` 字段由出入库模块在检验完成后通过领域事件回写，不在两处独立维护。

### 5.1 到货流程

```
采购订单（已下发）
    │
    ▼
创建到货通知单（ASN）
    │  填写：到货日期、运输方式、各物料到货数量
    ▼
仓库收货
    │  扫码/核对 → 实收数量登记
    │  可能部分到货（一次到一部分）
    ▼
来料检验（QC）
    │  面料：色差、克重、门幅、缩水率、色牢度
    │  辅料：规格、颜色、数量、外观
    │  包材：尺寸、印刷、材质
    ▼
    ├── 合格 → 入库（更新库存，采购订单行 delivered_quantity + accepted_quantity 更新）
    └── 不合格 → 退货/让步接收
         │
         ├── 退货 → 生成退货单，供应商补货或扣款
         └── 让步接收 → 标记质量等级，降级入库，记录偏差
```

## 十、MRP计算的注意事项

### 10.1 计算性能

MRP计算可能涉及大量数据（几十个生产订单 × 每个几十个BOM行），需要注意：

- BOM展开是纯计算，不涉及数据库写操作，可以放内存里算
- 库存查询是瓶颈，建议批量查询（一次查出所有相关物料的库存），不要循环单条查
- 计算结果写入数据库时用批量插入
- 大批量MRP可以考虑异步计算（前端展示计算进度）

### 10.2 重复计算幂等

同一个生产订单可能多次触发MRP计算，需要处理：

- 每次计算生成新的 batch_no
- 已 CONVERTED 的结果不受新计算影响
- 新计算只处理 PENDING 状态的结果
- 如果需求已变化（生产订单数量改了），旧的 PENDING 结果标记为 EXPIRED

### 10.3 面料色号匹配

MRP合并需求时，面料的合并条件不仅是 material_id，还要考虑颜色：

- 同款黑色风衣和卡其色风衣，虽然用同一种面料，但**颜色不同不能合并采购**
- 黑色款需要买黑色面料，卡其色款需要买卡其色面料
- 实际上在物料主数据里，不同颜色的面料应该就是不同的 material（黑色全棉斜纹布和卡其色全棉斜纹布是两个物料）
- 如果你的物料管理是"面料基材+颜色"的组合，合并时需要按 material_id 分组即可
