# 订单管理模块（Order Management）详细设计

> 版本：v1.0  
> 日期：2026-04-30  
> 适用行业：服装生产销售  
> 前置依赖：基础数据模块（SPU/SKU、客户、编码规则、季节）

---

## 一、模块总览

订单管理是整个系统的业务起点和枢纽。服装行业的订单有两个核心特征：

1. **以尺码矩阵为核心交互方式** — 订单不是按SKU逐行录入，而是按款式+颜色以矩阵形式填数量
2. **销售订单驱动生产订单** — 服装行业典型的"以销定产"模式，生产订单从销售订单转化而来

本模块包含：
1. 销售订单管理
2. 生产订单管理
3. 订单状态机
4. 尺码矩阵存储模型
5. 订单定价与折扣
6. 订单变更与跟踪

---

## 二、核心设计决策

### 决策1：订单行存的到底是什么？

**方案A：每个SKU一行** — 一个款3色5码15个SKU，订单里就有15行
**方案B：每个颜色款(Color-way)一行，数量存为矩阵JSON** — 一个款3色，订单里只有3行

**选择方案B。** 原因：

- 服装行业用户的思维方式是"这个款黑色要多少、卡其色要多少"，不是"黑色S码要多少、黑色M码要多少"
- 方案A在矩阵视图中需要15行数据拼装成矩阵展示，查询和渲染都更复杂
- 方案B的矩阵JSON天然对应前端的尺码矩阵组件，读写都很直观
- 统计时按行聚合颜色维度数量也很方便（一行就是一个颜色的总需求）

### 决策2：销售订单和生产订单的关系？

服装行业有两种模式：
- **以销定产**：先接客户订单，再安排生产 → 生产订单从销售订单转化
- **备货生产**：根据预测先生产，再销售 → 生产订单独立创建

系统必须同时支持两种模式，因此：

- 生产订单可以独立创建，也可以从销售订单转化
- 一张销售订单可以拆分为多张生产订单（不同款交期不同）
- 多张销售订单可以合并为一张生产订单（同款不同客户合并生产）
- 销售订单和生产订单之间是**多对多**关系

### 决策3：订单要不要做版本控制？

**要。** 服装行业订单变更频繁（客户改数量、改交期、追加订单），每次变更必须可追溯。

方案：不做全版本快照（太重），而是记录**变更流水** — 每次变更记录"哪个字段从什么值改成了什么值"，轻量且可追溯。

## 三、销售订单

### 3.1 状态机

```
                    ┌──────────┐
                    │  DRAFT   │ 草稿
                    └────┬─────┘
                         │ 提交
                    ┌────▼─────┐
                    │PENDING_AP│ 待审批
                    └────┬─────┘
                    ┌────┴────┐
                    │审批通过   │审批驳回
               ┌────▼────┐ ┌──▼──────┐
               │CONFIRMED │ │REJECTED │
               │ 已确认    │ │已驳回    │
               └────┬────┘ └────┬────┘
                    │           │修改后重新提交
                    │      ┌────▼─────┐
                    │      │PENDING_AP│
                    │      └─────────┘
              ┌─────┘
              │          │ 排产
              │     ┌────▼─────┐
              │     │PRODUCING │ 生产中
              │     └────┬─────┘
              │          │ 完工入库
              │     ┌────▼─────┐
              │     │ READY    │ 备货完成（库存已就绪）
              │     └────┬─────┘
              │          │ 发货
              │     ┌────▼─────┐
              │     │ SHIPPED  │ 已发货
              │     └────┬─────┘
              │          │ 签收
              │     ┌────▼─────┐
              │     │COMPLETED │ 已完成
              │     └──────────┘
              │
        ┌─────▼─────┐
        │ CANCELLED │ 已取消
        └───────────┘
```

**状态流转规则：**

| 当前状态 | 允许操作 | 目标状态 | 前置条件 |
|---------|---------|---------|---------|
| DRAFT | 提交 | PENDING_APPROVAL | 至少有一行明细 |
| DRAFT | 取消 | CANCELLED | 无 |
| PENDING_APPROVAL | 审批通过 | CONFIRMED | 审批引擎校验通过 |
| PENDING_APPROVAL | 审批驳回 | REJECTED | 无 |
| REJECTED | 修改后重新提交 | PENDING_APPROVAL | 无 |
| CONFIRMED | 开始排产 | PRODUCING | 已关联生产订单 |
| CONFIRMED | 取消 | CANCELLED | 未关联生产订单 |
| PRODUCING | 完工入库 | READY | 所有SKU库存满足 |
| PRODUCING | 部分发货 | SHIPPED | 允许部分发货时 |
| READY | 发货 | SHIPPED | 已创建出库单 |
| SHIPPED | 确认签收 | COMPLETED | 无 |
| — | — | — | 不设暂停状态；需要暂缓时在订单备注中标注原因，状态不变更 |

**关键约束：**
- 只有 DRAFT 状态允许编辑订单内容（数量、价格等）
- PENDING_APPROVAL 状态等待审批，不可编辑
- REJECTED 状态可修改后重新提交，回到 PENDING_APPROVAL
- CONFIRMED 之后只能变更交期和备注，不能改数量（改数量走变更单流程）
- CANCELLED 是终态，不可恢复
- 每次状态变更必须记录操作人、操作时间、变更原因

### 3.3 尺码矩阵存储结构

`size_matrix` 字段的 JSONB 结构：

```json
{
  "sizeGroupId": 1,
  "sizes": [
    {"sizeId": 10, "code": "S",  "quantity": 100},
    {"sizeId": 11, "code": "M",  "quantity": 200},
    {"sizeId": 12, "code": "L",  "quantity": 300},
    {"sizeId": 13, "code": "XL", "quantity": 200},
    {"sizeId": 14, "code": "XXL","quantity": 100}
  ],
  "totalQuantity": 900
}
```

**前端交互流程：**
1. 用户选择款式 → 系统加载该款式的颜色列表
2. 用户选择一个颜色 → 系统加载该款式关联的尺码组
3. 渲染矩阵：颜色为行（每行一个 order_sales_line），尺码为列
4. 用户在矩阵单元格中填写数量
5. 保存时，每行数据的 size_matrix 存对应尺码的数量

**查询优势：**
- 按订单查询所有行 → 按spu_id分组 → 天然还原出完整的尺码矩阵视图
- 按SKU汇总需求：`SELECT line.spu_id, line.color_way_id, size->>'sizeId', SUM((size->>'quantity')::int) FROM order_sales_line, jsonb_array_elements(size_matrix->'sizes') AS size GROUP BY ...`

### 3.4 销售订单与SKU的汇总视图

为了方便库存匹配和MRP运算，需要一个SKU级别的汇总视图：

```
order_sales_sku_summary（视图，非物理表）
├── order_id              BIGINT      订单ID
├── spu_id                BIGINT      款式ID
├── color_way_id          BIGINT      颜色款ID
├── sku_id                BIGINT      SKU ID（从size_matrix的sizeId关联md_sku）
├── quantity              INTEGER     需求数量
├── fulfilled_quantity    INTEGER     已满足数量（从出库记录汇总）
├── pending_quantity      INTEGER     待满足数量
└──
```

这个视图通过 SQL 函数从 `order_sales_line` 的 `size_matrix` 中拆解生成，后续采购和库存模块直接查这个视图。

## 四、生产订单

### 4.1 状态机

```
                    ┌──────────┐
                    │  DRAFT   │ 草稿
                    └────┬─────┘
                         │ 下达
                    ┌────▼─────┐
                    │ RELEASED │ 已下达（待排产）
                    └────┬─────┘
                         │ 排产
                    ┌────▼─────┐
                    │ PLANNED  │ 已排产
                    └────┬─────┘
                         │ 开始裁剪
                    ┌────▼─────┐
                    │ CUTTING  │ 裁剪中
                    └────┬─────┘
                         │ 进入缝制
                    ┌────▼─────┐
                    │ SEWING   │ 缝制中
                    └────┬─────┘
                         │ 后整
                    ┌────▼─────┐
                    │FINISHING │ 后整中
                    └────┬─────┘
                         │ 完工
                    ┌────▼─────┐
                    │COMPLETED │ 已完工
                    └────┬─────┘
                         │ 入库确认
                    ┌────▼─────┐
                    │ STOCKED  │ 已入库
                    └──────────┘
```

**为什么生产订单的状态比销售订单细？**

因为服装生产的核心流程就是：裁剪 → 缝制 → 后整。这是工厂车间管理的刚需 — 车间主任需要知道哪些款在裁剪、哪些在缝制、哪些快做完了。不需要上完整的MES系统，但至少要让订单在这个粒度上可跟踪。

**状态流转的灵活处理：**

不是所有服装都要走完整流程。比如：
- 针织衫可能没有裁剪环节（织造直接成型），可以从 PLANNED 直接到 SEWING
- 外购成衣（买手款）不需要生产，直接跳到 STOCKED

解决方案：**状态机可配置**。为每个订单的每个行项目标记"跳过裁剪"等标记，状态流转时根据标记决定是否跳过某个环节。但这个标记也要记在变更日志里。

### 4.3 生产订单行独立状态

**关键设计：生产订单的每一行有独立的状态。** 

因为同一个生产订单里，可能"黑色风衣已经在缝制了，卡其色风衣还在裁剪"。如果只有主表状态，无法反映这种差异。

主表状态取所有行的**最滞后的状态**。例如：3行分别是 SEWING、CUTTING、RELEASED → 主表状态为 RELEASED。

## 五、订单转化逻辑

### 5.1 销售订单 → 生产订单的转化流程

```
销售订单（已确认）
    │
    ▼
选择需要排产的款式行 ────► 检查：是否有未满足的库存？
    │                              │
    │                         ┌────┴────┐
    │                         │有库存    │无库存
    │                         ▼         ▼
    │                    锁定库存    创建生产订单
    │                    分配出库    关联销售订单
    │                         │         │
    │                         └────┬────┘
    │                              ▼
    │                     更新销售订单状态→PRODUCING
    ▼
未选中的行保持原状态，后续可再次转化
```

**具体规则：**

1. 用户在已确认的销售订单中选择一个或多个行，点击"生成生产订单"
2. 系统按 spu_id 汇总相同款式的需求（不同客户的同款需求可以合并生产）
3. 对于每个款式，检查当前可用库存是否满足：
   - 全部满足 → 不生成生产订单，直接锁定库存，创建预留出库单
   - 部分满足 → 锁定可用库存，不足部分生成生产订单
   - 完全不满足 → 全部生成生产订单
4. 生成生产订单时，自动关联 BOM（从 SPU 的 default_bom_id 取）
5. 建立 order_production_source 关联记录

### 5.2 多张销售订单合并生产

场景：客户A订了黑色风衣100件，客户B订了黑色风衣150件，可以合并为一张生产订单生产250件。

**操作方式：**
- 用户选择多张销售订单的行 → 批量"生成生产订单"
- 系统按 spu_id + color_way_id 分组汇总
- 同组的需求合并为一个生产订单行
- order_production_source 记录每个来源的分配数量

### 5.3 一张销售订单拆分生产

场景：一张销售订单有5个款，其中3个款在A车间生产，2个款在B车间生产。

**操作方式：**
- 用户分两次选择行，每次生成一张生产订单，指定不同车间
- 也可以一次全选，手动拆分到不同生产订单

## 六、订单定价与折扣

### 6.1 定价模型

服装行业的定价层次：

```
目录价（吊牌价）
  │
  ├── 批发价（经销商，通常为吊牌价的3-5折）
  │     │
  │     ├── 客户等级折扣（A客户95折，B客户9折...）
  │     │     │
  │     │     ├── 促销折扣（季末清仓等）
  │     │     │
  │     │     └── 最终成交价
  │     │
  │     └── ...
  │
  └── 零售价（直营店/线上，通常为吊牌价）
```

**设计原则：** 系统只记录最终成交价和折扣过程，不强制每层都走。灵活性优先。

### 6.2 订单行定价字段

```
order_sales_line 的定价相关字段：

├── unit_price          DECIMAL(12,2)  成交单价（最终价格）
├── list_price          DECIMAL(12,2)  吊牌价/目录价（参考，可选）
├── wholesale_price     DECIMAL(12,2)  批发价（参考，可选）
├── discount_rate       DECIMAL(5,4)   折扣率（如0.95）
├── discount_amount     DECIMAL(12,2)  折扣金额
├── actual_amount       DECIMAL(12,2)  行实际金额 = total_quantity × unit_price - discount_amount
└──
```

**自动定价逻辑（可配置，不强求）：**
1. 选择客户+款式后，系统建议单价 = 该SKU的 wholesale_price
2. 根据客户等级自动应用折扣率（A客户95折、B客户9折等）
3. 用户可以手动覆盖单价和折扣率
4. 系统记录建议价和最终价，方便后续分析

### 6.3 客户等级价格表（可选功能）

```
order_customer_price（客户等级价格表）
├── id                       BIGPK
├── customer_level           VARCHAR(16)  客户等级（A/B/C/D）
├── category_id              BIGINT       品类ID（不同品类折扣不同）
├── discount_rate            DECIMAL(5,4) 折扣率
├── min_quantity             INTEGER      起订量（可选，达量折扣）
├── effective_from           DATE         生效日期
├── effective_to             DATE         失效日期
├── created_by / created_at / updated_by / updated_at / deleted / version
└──
```

## 七、订单变更与跟踪

### 7.1 订单变更记录

每次订单内容变更（不仅是状态变更），都要记录变更明细：

```
order_change_log（订单变更日志）
├── id                       BIGPK          主键
├── order_type               VARCHAR(16)    订单类型：SALES/PRODUCTION
├── order_id                 BIGINT         订单ID
├── order_line_id            BIGINT         订单行ID（NULL表示主表变更）
├── change_type              VARCHAR(16)    变更类型：STATUS_CHANGE/FIELD_CHANGE/LINE_ADD/LINE_REMOVE/QUANTITY_CHANGE
├── field_name               VARCHAR(64)    变更字段名（如 status, delivery_date, size_matrix）
├── old_value                TEXT           变更前值
├── new_value                TEXT           变更后值
├── change_reason            TEXT           变更原因
├── operated_by              BIGINT         操作人ID
├── operated_at              TIMESTAMP      操作时间
└──
```

**触发方式：**
- 状态变更：在状态机转换方法中自动写入
- 字段变更：通过领域事件或AOP拦截，对比变更前后值
- 行增删：在Service层显式记录

### 7.2 订单跟踪时间线

### 7.3 数量变更的特殊处理

已确认的订单如果需要改数量，不是直接改，而是走**变更单**流程：

```
order_quantity_change（数量变更单）
├── id                       BIGPK          主键
├── order_id                 BIGINT         原订单ID
├── order_line_id            BIGINT         原订单行ID
├── change_type              VARCHAR(16)    INCREASE/DECREASE
├── size_matrix_before       JSONB          变更前矩阵
├── size_matrix_after        JSONB          变更后矩阵
├── diff_matrix              JSONB          差异矩阵（变更前后差值）
├── reason                   TEXT           变更原因
├── status                   VARCHAR(16)    PENDING/APPROVED/REJECTED
├── approved_by              BIGINT         审批人
├── approved_at              TIMESTAMP      审批时间
├── created_by / created_at / updated_by / updated_at / deleted / version
└──
```

**为什么已确认订单改数量要这么重？** 因为数量变更会影响：
- 已关联的生产订单（需求量变了）
- 已锁定的库存（可能需要调整）
- 采购需求（面料辅料可能要追加采购）

所以必须走审批，审批通过后再联动调整下游。这是企业级和"增删改查"系统的核心区别。

## 八、订单与库存的交互

### 8.1 库存预留（Allocation）

销售订单确认后，不是立即扣库存，而是**预留库存**：

```
inventory_allocation（库存预留 — 完整定义见库存管理模块 05-库存管理模块-详细设计.md 第五节）
├── id                       BIGPK
├── order_type               VARCHAR(16)    SALES/PRODUCTION
├── order_id                 BIGINT         订单ID
├── order_line_id            BIGINT         订单行ID
├── sku_id                   BIGINT         SKU ID（成品预留时使用）
├── material_id              BIGINT         物料ID（原料预留时使用，领料场景）
├── warehouse_id             BIGINT         仓库ID
├── batch_no                 VARCHAR(32)    批次号（可选，指定批次预留）
├── allocated_qty            INTEGER        预留数量
├── fulfilled_qty            INTEGER        已出库数量
├── remaining_qty            INTEGER        剩余预留 = allocated - fulfilled
├── status                   VARCHAR(16)    ACTIVE/PARTIAL_FULFILLED/FULFILLED/RELEASED/EXPIRED
├── expire_at                TIMESTAMP      过期时间（到期自动释放）
├── remark                   TEXT
└──
```

**预留逻辑：**
- 销售订单 CONFIRMED → 对可用库存做预留（可用 → 锁定）
- 预留后：`可用库存 = 实际库存 - 锁定库存`
- 订单取消 → 释放预留（锁定 → 可用）
- 出库完成 → 预留状态变为 FULFILLED

### 8.2 库存满足率

前端订单详情页需要展示每个SKU的库存满足情况：

```
┌────────┬──────┬──────┬──────┬──────┬──────┐
│ 颜色    │  S   │  M   │  L   │  XL  │ XXL  │
├────────┼──────┼──────┼──────┼──────┼──────┤
│ 黑色    │100   │200   │300   │200   │100   │ ← 订单数量
│        │✅100 │✅150 │❌50  │✅200 │❌0   │ ← 可用库存
│        │     │⚠️-50│     │     │⚠️100│ ← 缺口
├────────┼──────┼──────┼──────┼──────┼──────┤
│ 卡其色  │ 50  │ 100  │ 150  │ 100  │  50  │
│        │✅80 │✅100 │✅200 │✅100 │✅60  │ ← 库存充足
└────────┴──────┴──────┴──────┴──────┴──────┘

✅ 满足  ⚠️ 部分满足/不满足  ❌ 缺货
```

这个视图需要实时计算，通过 SKU 汇总视图 + 库存模块的可用库存数据 JOIN 得到。

订单模块发布的领域事件，供其他模块订阅：

| 事件名 | 触发时机 | 订阅方 | 用途 |
|--------|---------|--------|------|
| SalesOrderConfirmed | 销售订单确认 | 库存模块 | 触发库存预留 |
| SalesOrderCancelled | 销售订单取消 | 库存模块 | 释放预留库存 |

| ProductionOrderCompleted | 生产订单完工 | 库存模块 | 准备入库 |
| ProductionOrderStocked | 生产订单入库完成 | 订单模块 | 通知销售订单可以发货 |
| OrderQuantityChanged | 订单数量变更审批通过 | 库存/采购模块 | 调整预留/采购需求 |

**事件实现方式：**
- 初期：Spring ApplicationEvent + Outbox发件箱（和业务数据同事务，保证不丢失）

```java
}
```

## 十二、退货管理

### 12.1 退货场景

| 退货类型 | 编码 | 触发场景 | 货物去向 |
|---------|------|---------|---------|
| 客户退货 | CUSTOMER_REJECT | 客户收货后因质量/尺码问题退回 | 质检→合格入可用库存/不合格报废 |
| 物流拒收 | LOGISTICS_REJECT | 发货后客户拒收或无法送达 | 原路退回→质检→入可用库存 |
| 经销商退货 | DISTRIBUTOR_RETURN | 经销商退回未售出的库存（换季退货） | 质检→合格入可用库存/不合格报废 |

### 12.2 退货流程

```
客户/经销商发起退货申请
    │
    ▼
业务员审核退货申请
    │  审核通过 → 生成退货入库单
    │  审核驳回 → 通知客户
    ▼
仓库收货（同采购收货流程）
    │  扫码确认 → 质检库存增加
    ▼
质检
    │  合格 → 可用库存增加
    │  不合格 → 标记报废/次品
    ▼
退货完成
    │  更新销售订单状态
    │  生成退款记录（交财务处理，本期仅记录）
```

### 12.3 表结构

```
order_return（退货单）
├── id                       BIGPK          主键
├── return_no                VARCHAR(32)    退货单号
├── return_type              VARCHAR(16)    CUSTOMER_REJECT / LOGISTICS_REJECT / DISTRIBUTOR_RETURN
├── sales_order_id           BIGINT         原销售订单ID
├── sales_order_no           VARCHAR(32)    原销售订单编号
├── customer_id              BIGINT         客户ID
├── reason                   TEXT           退货原因
├── status                   VARCHAR(16)    DRAFT/PENDING_APPROVAL/APPROVED/RECEIVING/QC/COMPLETED/REJECTED
├── total_quantity           INTEGER        退货总数量
├── inbound_order_id         BIGINT         关联的退货入库单ID
├── approved_by              BIGINT         审批人
├── approved_at              TIMESTAMP      审批时间
├── remark                   TEXT
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

order_return_line（退货单行）
├── id                       BIGPK          主键
├── return_id                BIGINT         退货单ID
├── sales_order_line_id      BIGINT         原销售订单行ID
├── spu_id                   BIGINT         款式ID
├── color_way_id             BIGINT         颜色款ID
├── size_matrix              JSONB          退货尺码矩阵（同订单结构，入库时按SKU粒度解析）
├── total_quantity           INTEGER        本行退货数量
├── qc_passed_qty            INTEGER        质检合格数量
├── qc_failed_qty            INTEGER        质检不合格数量
├── qc_result                JSONB          质检结果
├── remark                   TEXT
├── created_by / created_at / updated_by / updated_at / deleted / version
└──
```

**退货入库的 SKU 粒度处理：** `size_matrix` 存储尺码矩阵数据，退货入库时由系统解析 JSONB 将矩阵拆解为 SKU 级别的库存操作。具体逻辑：遍历 size_matrix.sizes，通过 (spu_id, color_way_id, sizeId) 关联 md_sku 获取 sku_id，逐个 SKU 执行 INBOUND_RETURN 库存操作。

### 12.4 退货与库存的交互

```
退货收货确认 → 库存操作：INBOUND_RETURN
    质检库存 += 收货数量

退货质检通过 → 库存操作：QC_PASS
    质检库存 -= 合格数量
    可用库存 += 合格数量

退货质检不通过 → 库存操作：QC_FAIL
    质检库存 -= 不合格数量
    （不合格品不走回可用库存，标记为报废/次品）
```
