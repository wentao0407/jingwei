# 基础数据模块（Master Data）详细设计

> 版本：v1.0  
> 日期：2026-04-30  
> 适用行业：服装生产销售  

---

## 一、模块总览

基础数据模块是整个系统的地基，为订单、采购、库存、出入库等业务模块提供统一的主数据服务。

核心子模块：
1. 物料主数据 — 面料、辅料、包材、成品
2. SPU/SKU 管理 — 款式→颜色→尺码的层级结构
3. 季节与波段管理
4. 尺码组管理
5. 编码规则引擎
6. 供应商/客户档案
7. 仓库/库位档案

## 二、物料主数据属性模型

### 2.1 核心挑战

服装行业的物料分为面料、辅料、包材三大类，每类有完全不同的属性集：

| 属性 | 面料 | 辅料 | 包材 |
|------|------|------|------|
| 克重(g/m²) | ✅ | ❌ | ❌ |
| 门幅(cm) | ✅ | ❌ | ❌ |
| 成分 | ✅ | ✅ | ❌ |
| 纱支 | ✅ | ❌ | ❌ |
| 规格 | ❌ | ✅ | ✅ |
| 材质 | ❌ | ✅ | ✅ |

> **注意：** 成品不走 `md_material` 表，而是走 SPU/SKU 模型（见下一节）。`md_material.type` 枚举仅包含 FABRIC/TRIM/PACKAGING 三种类型，不包含 PRODUCT。成品如需记录物料属性（如成分），通过 SPU 的扩展字段或关联 `md_material` 来处理。

如果每种物料建一张表，后期加新类型就要改表改代码；如果把所有属性放在一张宽表里，大部分字段都是空的。

### 2.2 方案：混合模型（公共字段 + JSONB 扩展属性）

**设计原则：**
- 高频查询、条件筛选、外键关联的属性 → 提取为独立列
- 差异化、低频查询的属性 → 存入 JSONB 扩展字段
- 属性定义本身也做数据化管理，方便前端动态渲染表单

**表结构示意：**

```
md_material（物料主表）
├── id                    BIGPK       主键
├── code                  VARCHAR(32) 物料编码（编码规则生成）
├── name                  VARCHAR(128)物料名称
├── type                  ENUM        物料类型：FABRIC/TRIM/PACKAGING
├── category_id           BIGINT      物料分类ID（外键）
├── unit                  VARCHAR(16) 基本单位（米/个/套/件）
├── status                ENUM        状态：ACTIVE/INACTIVE
├── ext_attrs             JSONB       扩展属性（按type不同存不同结构）
├── remark                TEXT        备注
├── created_by            BIGINT      创建人
├── created_at            TIMESTAMP   创建时间
├── updated_by            BIGINT      更新人
├── updated_at            TIMESTAMP   更新时间
├── deleted                BOOLEAN     软删除标记（DEFAULT FALSE）
└── version               INTEGER     乐观锁版本号
```

**ext_attrs 按 type 的存储约定：**

面料（FABRIC）：
```json
{
  "weight": 280,
  "weightUnit": "g/m²",
  "width": 150,
  "widthUnit": "cm",
  "composition": [
    {"fiber": "棉", "percentage": 80},
    {"fiber": "涤纶", "percentage": 20}
  ],
  "yarnCount": "40S/1",
  "weaveType": "斜纹",
  "shrinkage": 3,
  "colorFastness": "4级"
}
```

辅料（TRIM）：
```json
{
  "spec": "4孔 18mm",
  "material": "树脂",
  "color": "黑色",
  "finishingType": "哑光",
  "applicableStyle": "通用"
}
```

包材（PACKAGING）：
```json
{
  "spec": "60×40×30cm",
  "material": "瓦楞纸",
  "thickness": "5层",
  "loadBearing": "15kg"
}
```

> **注意：** 成品统一通过 SPU/SKU 模型管理，不在 `md_material` 表中存储。详见下一节。

### 2.3 属性定义管理

为了让前端能根据物料类型动态渲染不同的属性表单，需要一张属性定义表：

```
md_attribute_def（属性定义）
├── id                    BIGPK       主键
├── code                  VARCHAR(32) 属性编码（如 fabric_weight）
├── name                  VARCHAR(64) 属性名称（如 面料克重）
├── material_type         ENUM        适用物料类型
├── input_type            ENUM        输入类型：TEXT/NUMBER/SELECT/MULTI_SELECT/COMPOSITION
├── required              BOOLEAN     是否必填
├── sort_order            INTEGER     排序号
├── options               JSONB       选项列表（SELECT类型用）
├── ext_json_path         VARCHAR(64) JSONB中对应路径（如 weight）
└── ...
```

**好处：**
- 前端根据 material_type 查询属性定义，动态渲染表单
- 新增属性不需要改表结构，加一条配置数据就行
- ext_json_path 告诉后端把这个属性值写到 JSONB 的哪个路径下

## 三、SPU/SKU 管理模型

### 3.1 三层结构

```
SPU（款式）
 └── Color-way（颜色款）
      └── SKU（最小库存单元 = 颜色 + 尺码）
```

**SPU**：描述一个"款式"的共性信息
- 款式名称：2026春季女士风衣
- 款式编号：SP20260001
- 季节：2026春
- 品类：女士外套/风衣
- 设计稿/款式图
- 默认BOM

**Color-way**：款式下的颜色维度
- 颜色名称：黑色
- 颜色编码：BK
- 色号/潘通号（可选）
- 对应面料（不同颜色可能用不同面料）

**SKU**：最终参与库存、订单、出入库的最小单元
- 编码/条码：SP20260001-BK-M
- 成本价
- 销售价
- 库存数量（从库存模块实时读取，不在此表存储）

### 3.2 表结构设计

```
md_spu（款式主表）
├── id                    BIGPK       主键
├── code                  VARCHAR(32) 款式编码
├── name                  VARCHAR(128)款式名称
├── season_id             BIGINT      季节ID（外键）
├── category_id           BIGINT      品类ID（外键）
├── brand_id              BIGINT      品牌ID（外键，可选）
├── size_group_id         BIGINT      尺码组ID（外键，决定有哪些尺码）
├── design_image          VARCHAR(256)款式图URL
├── default_bom_id        BIGINT      默认BOM ID（外键，可选）
├── status                ENUM        状态：DRAFT/ACTIVE/INACTIVE
├── remark                TEXT        备注
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

md_color_way（颜色款）
├── id                    BIGPK       主键
├── spu_id                BIGINT      款式ID（外键）
├── color_name            VARCHAR(32) 颜色名称
├── color_code            VARCHAR(16) 颜色编码（用于SKU编码拼接）
├── pantone_code          VARCHAR(16) 潘通色号（可选）
├── fabric_material_id    BIGINT      对应面料ID（外键，不同颜色可能不同面料）
├── color_image           VARCHAR(256)颜色款图片URL（可选）
├── sort_order            INTEGER     排序号
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

md_sku（最小库存单元）
├── id                    BIGPK       主键
├── code                  VARCHAR(64) SKU编码（自动生成，如 SP20260001-BK-M）
├── barcode               VARCHAR(64) 条码（可自动生成或外部导入）
├── spu_id                BIGINT      款式ID（冗余，方便查询）
├── color_way_id          BIGINT      颜色款ID（外键）
├── size_id               BIGINT      尺码ID（外键）
├── cost_price            DECIMAL(12,2)成本价
├── sale_price            DECIMAL(12,2)销售价
├── wholesale_price       DECIMAL(12,2)批发价（可选）
├── status                ENUM        状态：ACTIVE/INACTIVE
├── created_by / created_at / updated_by / updated_at / deleted / version
└──
```

### 3.3 SKU 自动生成规则

创建 SPU 时，用户选定：
1. 颜色列表（如：黑色、卡其色、藏青色）
2. 尺码组（如：S/M/L/XL/XXL）

系统自动交叉生成所有 SKU：
```
黑色-S, 黑色-M, 黑色-L, 黑色-XL, 黑色-XXL
卡其色-S, 卡其色-M, 卡其色-L, 卡其色-XL, 卡其色-XXL
藏青色-S, 藏青色-M, 藏青色-L, 藏青色-XL, 藏青色-XXL
= 3×5 = 15 个 SKU
```

**生成流程：**
1. 用户填写 SPU 基础信息，选择颜色和尺码组
2. 系统创建 SPU 记录
3. 系统为每个颜色创建 Color-way 记录
4. 系统按颜色×尺码交叉生成 SKU 记录，编码自动拼接
5. 用户在矩阵视图中补充各 SKU 的价格（可批量设置）

**新增颜色或尺码：** SPU 创建后，可以追加颜色或尺码组中新增尺码，系统增量生成对应的 Color-way 和 SKU，不影响已有数据。

## 六、编码规则引擎

### 6.1 设计思路

所有业务单据的编号必须支持可配置的编码规则，这是企业级系统的标配。

编码规则由多个**段（Segment）**拼接而成，每段有固定类型：

| 段类型 | 说明 | 示例 |
|--------|------|------|
| FIXED | 固定文本 | SO、RK、CK |
| DATE | 日期格式 | 20260430、202604 |
| SEQUENCE | 流水号 | 0001、00001 |
| SEASON | 季节编码 | 26SS |
| WAREHOUSE | 仓库编码 | WH01 |
| CUSTOM | 自定义值 | 用户输入 |

### 6.2 规则配置示例

**销售订单编号规则：** SO + 年月 + 5位流水号
```
SO-202604-00001
├── FIXED: "SO-"
├── DATE: "YYYYMM"
└── SEQUENCE: 5位，按月重置
```

**入库单编号规则：** RK + 仓库编码 + 年月日 + 4位流水号
```
RK-WH01-20260430-0001
├── FIXED: "RK-"
├── WAREHOUSE: 当前仓库编码
├── DATE: "-YYYYMMDD-"
└── SEQUENCE: 4位，按日按仓库重置
```

**SKU编码规则：** 款式编码 + 颜色编码 + 尺码编码
```
SP20260001-BK-M
├── CUSTOM: 款式编码
├── CUSTOM: 颜色编码
└── CUSTOM: 尺码编码
```

### 6.3 表结构

```
md_coding_rule（编码规则）
├── id                    BIGPK       主键
├── code                  VARCHAR(32) 规则编码（如 ORDER_CODE、INBOUND_CODE）
├── name                  VARCHAR(64) 规则名称
├── business_type         VARCHAR(32) 业务类型
├── description           TEXT        说明
├── status                ENUM        ACTIVE/INACTIVE
├── created_by / created_at / updated_by / updated_at / deleted / version
└──

md_coding_rule_segment（编码规则段）
├── id                    BIGPK       主键
├── rule_id               BIGINT      规则ID
├── segment_type          ENUM        FIXED/DATE/SEQUENCE/SEASON/WAREHOUSE/CUSTOM
├── segment_value         VARCHAR(64) 段值（FIXED类型填固定文本，DATE类型填格式如YYYYMM）
├── seq_length            INTEGER     流水号长度（SEQUENCE类型专用）
├── seq_reset_type        ENUM        重置方式：NEVER/YEARLY/MONTHLY/DAILY（SEQUENCE类型专用）
├── connector             VARCHAR(4)  连接符（拼接时本段与前段之间，如"-"或""）
├── sort_order            INTEGER     排序号
└──

md_coding_sequence（编码流水号）
├── id                    BIGPK       主键
├── rule_id               BIGINT      规则ID
├── reset_key             VARCHAR(32) 重置键（如 202604 表示按月重置时的当前月份）
├── current_value         BIGINT      当前值
├── updated_at            TIMESTAMP   更新时间
└──
```

### 6.4 生成逻辑（伪代码）

```java
public String generateCode(String ruleCode, Map<String, String> context) {
    // 1. 查询规则及段列表
    CodingRule rule = ruleRepo.findByCode(ruleCode);
    List<CodingRuleSegment> segments = segmentRepo.findByRuleId(rule.getId());

    StringBuilder code = new StringBuilder();

    for (CodingRuleSegment seg : segments) {
        // 拼接连接符
        code.append(seg.getConnector());

        switch (seg.getSegmentType()) {
            case FIXED:
                code.append(seg.getSegmentValue());
                break;
            case DATE:
                code.append(LocalDate.now().format(
                    DateTimeFormatter.ofPattern(seg.getSegmentValue())
                ));
                break;
            case SEQUENCE:
                String resetKey = buildResetKey(seg.getSeqResetType());
                Long nextVal = sequenceRepo.incrementAndGet(
                    rule.getId(), resetKey
                );
                code.append(String.format(
                    "%0" + seg.getSeqLength() + "d", nextVal
                ));
                break;
            case WAREHOUSE:
                code.append(context.get("warehouseCode"));
                break;
            case SEASON:
                code.append(context.get("seasonCode"));
                break;
            case CUSTOM:
                code.append(context.get(seg.getSegmentValue()));
                break;
        }
    }

    return code.toString();
}
```

**并发安全：** 流水号递增必须用数据库行级锁（`SELECT ... FOR UPDATE`）或 Redis INCR，保证不会重复。在高并发场景下优先用 Redis INCR。

- PICKING：拣货位，高频出入
- STAGING：暂存位，出入库中间态

## 十一、公共字段约定

所有业务表统一包含以下审计字段，后续不再重复列出：

| 字段 | 类型 | 说明 |
|------|------|------|
| created_by | BIGINT | 创建人ID |
| created_at | TIMESTAMP | 创建时间 |
| updated_by | BIGINT | 最后修改人ID |
| updated_at | TIMESTAMP | 最后修改时间 |
| deleted | BOOLEAN | 软删除标记，默认FALSE |
| version | INTEGER | 乐观锁版本号，@Version |

**统一行为：**
- 所有查询默认加 `WHERE deleted = FALSE`
- 更新操作用乐观锁：`UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?`
- 删除操作只做 `UPDATE SET deleted = TRUE`，永远不物理删除
- 审计字段由框架层自动填充（MyBatis-Plus 的 MetaObjectHandler 或 JPA AuditorAware）
