> 版本：v1.0  
> 日期：2026-04-30  
> 项目代号：JingWei  
> 包命名：com.jingwei  

---

## 一、概述

### 1.1 为什么需要专门的一致性方案？

经纬系统涉及订单、采购、库存、仓储四大业务域，数据在模块间流转。如果一致性保障不到位，会出现：

- 库存超卖 — 客户下单了但没货发
- 事件丢失 — 生产订单创建了但采购没算
- 重复操作 — 收货执行两次，库存翻倍
- 数据不一致 — 出库单已发货但库存没扣

这类问题在开发和测试中很难复现，但上线后在并发和生产环境下必然暴露，且修复代价极大。因此一致性保障必须从架构层面系统性解决，而非事后打补丁。

### 1.2 一致性原则

| 原则 | 说明 |
|------|------|
| 模块内强一致 | 同一模块内的操作用数据库事务保证原子性 |
| 跨模块最终一致 | 跨模块操作通过领域事件+Outbox保证最终一致 |
| 业务容忍优先 | 采购建议、MRP计算等允许基于稍旧数据，不做强一致 |
| 所有操作可追溯 | 流水记录before/after，任何时刻的库存状态可重建 |
| 兜底对账不可少 | 预防性方案不能100%，定期对账是最后防线 |

## 二、一致性挑战分析

### 2.1 挑战总览

| 编号 | 场景 | 风险 | 严重程度 |
|------|------|------|:--------:|
| C-01 | 库存并发扣减 | 超卖/超出 | 🔴 高 |
| C-02 | 跨模块操作中途失败 | 数据不一致 | 🔴 高 |
| C-03 | 领域事件丢失 | 下游模块漏处理 | 🔴 高 |
| C-04 | 出库单与库存状态不一致 | 库存数据不准 | 🟡 中 |
| C-05 | 重复操作 | 数据翻倍 | 🟡 中 |
| C-06 | 长计算期间数据变动 | 计算结果偏差 | 🟢 低 |
| C-07 | 编码流水号重复 | 单据号冲突 | 🟡 中 |
| C-08 | 跨仓库调拨中间态 | 两仓数据对不上 | 🟡 中 |

### 2.2 场景详解

**C-01 库存并发扣减**

```
9:00:00.000  销售订单A需要黑色M码60件 → 查可用库存=100 → 够
9:00:00.050  销售订单B需要黑色M码50件 → 查可用库存=100 → 够
9:00:00.100  订单A预留执行：available -= 60, locked += 60
9:00:00.120  订单B预留执行：available -= 50, locked += 50
9:00:00.150  结果：available = -10, locked = 110 ❌
```

风险：两个事务读到同一时刻的库存值（100），各自扣减后总锁定量超过实际库存。

**C-02 跨模块操作中途失败**

```
销售订单确认：
  Step 1: UPDATE order_sales SET status='CONFIRMED' ✅
  Step 2: 库存预留（可用→锁定）❌ 此时空指针异常

结果：订单已确认，但库存未预留
影响：其他订单看不到这些库存被占，可能导致超卖
```

**C-03 领域事件丢失**

```
生产订单创建：
  Step 1: 保存生产订单到数据库 ✅
  Step 2: 发布ProductionOrderCreatedEvent → 内存事件总线
  Step 3: JVM在发布后0.01秒崩溃 💥

结果：生产订单已存在，但采购模块未收到事件，MRP未计算
影响：面料辅料未采购，生产停滞
```

**C-04 出库单与库存状态不一致**

```
发货确认：
  Step 1: 更新出库单状态 → SHIPPED ✅
  Step 2: 扣减锁定库存 ❌ 数据库连接超时

结果：出库单显示已发货，但库存未扣减
影响：库存数据虚高，后续MRP计算和库存查询都不准
```

**C-05 重复操作**

```
场景1：仓库人员手抖，连续点击"确认收货"两次
  → 收货操作执行两次 → 质检库存多加一倍

场景2：前端网络超时，自动重试
  → 同一个出库确认请求发两次 → 库存扣两次

场景3：MQ消息重复投递
  → 同一个GoodsAcceptedEvent被消费两次 → 可用库存加两次
```

**C-06 MRP计算期间数据变动**

```
MRP计算开始（14:30:00）：
  查询面料可用库存 = 800m

计算过程中（14:30:02）：
  另一笔领料出库扣减了300m

MRP计算结束（14:30:05）：
  基于800m算出净需求 = 200m
  但实际此时库存只剩500m，净需求应为500m
```

风险等级低，因为MRP结果是"建议"而非"直接扣库存"，后续需人工审核。

**C-07 编码流水号重复**

```
两个并发请求同时生成入库单号：
  请求A：读取当前流水号=0001 → 生成 RK-20260430-0001
  请求B：读取当前流水号=0001 → 生成 RK-20260430-0001 ❌
```

**C-08 跨仓库调拨中间态**

```
调拨 WH01 → WH02，100件黑色M码：
  Step 1: WH01 available -= 100 ✅
  Step 2: 货物在运输中（两仓都不算）
  Step 3: WH02 available += 100 → 如果Step 3失败，100件"消失"
```

## 三、第一层：预防性保障（事前）

### 3.1 乐观锁 — 解决C-01

**适用场景：** 库存扣减、预留等并发操作

**原理：** 每条记录带 `version` 字段，更新时校验版本号，冲突则重试。

**实现：**

```java
/**
 * 库存变更核心方法 — 所有库存操作必须经过此方法
 */
@Transactional
public void changeInventory(InventoryChangeRequest request) {
    // 1. 加载库存记录（含version）
    InventoryRecord record = inventoryRepo.findById(request.getInventoryId());
    
    // 2. 校验操作合法性（可用库存不可为负等）
    validateChange(request, record);
    
    // 3. 执行变更
    applyChange(record, request);
    
    // 4. 保存（乐观锁在这里生效）
    //    SQL: UPDATE ... SET version = version + 1 WHERE id = ? AND version = ?
    int affected = inventoryRepo.saveWithVersion(record);
    
    if (affected == 0) {
        throw new OptimisticLockConflictException(
            "库存记录已被其他操作修改，请重试");
    }
    
    // 5. 写操作流水
    operationRepo.insert(buildOperation(request, record));
}
```

**冲突重试策略：**

```java
/**
 * 带重试的库存操作
 */
public void changeInventoryWithRetry(InventoryChangeRequest request) {
    int maxRetry = 3;
    for (int i = 0; i < maxRetry; i++) {
        try {
            changeInventory(request);
            return; // 成功
        } catch (OptimisticLockConflictException e) {
            if (i == maxRetry - 1) {
                throw new ConcurrentConflictException(
                    "库存操作冲突，已重试" + maxRetry + "次仍失败，请稍后重试");
            }
            // 退避等待：50~150ms随机，避免重试风暴
            Thread.sleep(50 + ThreadLocalRandom.current().nextInt(100));
        }
    }
}
```

**热点降级 — Redis原子扣减：**

当某个SKU的乐观锁冲突率超过阈值（如连续3次冲突），自动降级为Redis原子操作：

```java
/**
 * 热点SKU库存扣减 — Redis原子操作
 */
public void allocateByRedis(Long skuId, Long warehouseId, Integer qty) {
    String key = "inventory:available:" + skuId + ":" + warehouseId;
    
    // Redis INCRBY 是原子的，不会超扣
    Long remaining = redisTemplate.opsForValue().increment(key, -qty);
    
    if (remaining < 0) {
        // 库存不足，回滚
        redisTemplate.opsForValue().increment(key, qty);
        throw new InsufficientInventoryException(skuId, warehouseId);
    }
    
    // Redis扣减成功，异步同步到数据库
    // 这里即使同步失败，Redis的数据是准的，可通过日终对账修正
    eventPublisher.publish(new RedisInventorySyncEvent(skuId, warehouseId, qty));
}
```

**Redis与数据库的一致性：**

```
写入路径：业务操作 → Redis原子扣减 → 异步写数据库
读取路径：先查Redis → Redis无数据则查数据库并回填Redis

兜底：日终对账（见第四章）
```

### 3.2 数据库事务 — 解决C-02、C-04

**适用场景：** 模块内或跨模块（同库）的多步操作

**核心前提：** 模块化单体架构下所有模块共享一个PostgreSQL实例，`@Transactional` 能覆盖跨模块操作。

**事务边界设计原则：**

| 原则 | 说明 | 反例 |
|------|------|------|
| 事务尽量短 | 避免长事务占用连接和锁 | MRP计算不用大事务包裹 |
| 事务内不做外部调用 | 不在事务内调用外部API或发MQ | 事务内调用物流API ❌ |
| 只写必要的数据 | 变更日志和业务操作同事务，发事件走Outbox | 事件直接在事务内发布 ❌ |

**关键事务场景：**

**场景1：销售订单确认（跨订单+库存模块）**

```java
@Transactional
public void confirmSalesOrder(Long orderId) {
    // Step 1: 状态机校验 + 更新订单状态
    SalesOrder order = orderRepo.findById(orderId);
    SalesOrderStatus newStatus = stateMachine.fireEvent(
        order.getStatus(), SUBMIT, context);
    order.setStatus(newStatus);
    orderRepo.save(order);     // 乐观锁
    
    // Step 2: 库存预留（同库同事务）
    for (SalesOrderLine line : order.getLines()) {
        List<SkuQuantity> skuQties = line.getSizeMatrix().toSkuQuantities();
        for (SkuQuantity sq : skuQties) {
            if (sq.getQuantity() > 0) {
                inventoryDomainService.changeInventoryWithRetry(
                    InventoryChangeRequest.builder()
                        .operationType(OperationType.ALLOCATE)
                        .skuId(sq.getSkuId())
                        .warehouseId(order.getWarehouseId())
                        .quantity(sq.getQuantity())
                        .sourceType("SALES_ORDER")
                        .sourceId(orderId)
                        .build()
                );
            }
        }
    }
    
    // Step 3: 创建预留记录
    for (...) {
        allocationRepo.save(new InventoryAllocation(...));
    }
    
    // Step 4: 写变更日志
    changeLogService.log(orderId, null, "STATUS_CHANGE", 
        "status", "DRAFT", "CONFIRMED", "SUBMIT", operatorId);
    
    // Step 5: 领域事件写入Outbox（同事务）
    outboxRepo.save(DomainEvent.of(
        "SalesOrderConfirmed", orderId, order.toEventPayload()));
    
    // 任何一步抛异常 → 全部回滚，订单回到DRAFT，库存不扣
}
```

**场景2：发货确认（跨出库+库存+订单模块）**

```java
@Transactional
public void confirmShipment(Long outboundId) {
    // Step 1: 更新出库单状态
    OutboundOrder outbound = outboundRepo.findById(outboundId);
    outbound.setStatus(SHIPPED);
    outboundRepo.save(outbound);
    
    // Step 2: 扣减锁定库存
    for (OutboundLine line : outbound.getLines()) {
        inventoryDomainService.changeInventoryWithRetry(
            InventoryChangeRequest.builder()
                .operationType(OperationType.OUTBOUND_SALES)
                .skuId(line.getSkuId())
                .quantity(line.getActualQty())
                .build()
        );
    }
    
    // Step 3: 更新预留记录
    for (OutboundLine line : outbound.getLines()) {
        allocationRepo.fulfill(line.getAllocationId(), line.getActualQty());
    }
    
    // Step 4: 更新销售订单状态
    salesOrderDomainService.onOutboundShipped(outbound.getSalesOrderId());
    
    // Step 5: 事件Outbox
    outboxRepo.save(DomainEvent.of("OutboundShipped", outboundId, ...));
}
```

### 3.3 事务性发件箱（Transactional Outbox）— 解决C-03

**适用场景：** 跨模块领域事件传递

**问题本质：** 业务数据写入数据库和事件发布到内存不是原子操作，中间崩溃事件就丢了。

**解决方案：** 把事件也写入数据库（Outbox表），和业务操作在同一个事务中提交。

**Outbox表结构：**

```sql
CREATE TABLE domain_event_outbox (
    id              BIGSERIAL PRIMARY KEY,
    event_id        VARCHAR(36)  NOT NULL,     -- UUID，全局唯一
    event_type      VARCHAR(64)  NOT NULL,     -- ProductionOrderCreated
    aggregate_type  VARCHAR(32)  NOT NULL,     -- PRODUCTION_ORDER
    aggregate_id    BIGINT       NOT NULL,
    payload         JSONB        NOT NULL,     -- 事件数据
    published       BOOLEAN      DEFAULT FALSE,
    created_at      TIMESTAMP    DEFAULT NOW(),
    published_at    TIMESTAMP
);

-- 只查未发布的事件，高效
CREATE INDEX idx_outbox_unpublished ON domain_event_outbox (created_at) 
    WHERE published = FALSE;

-- 已发布事件的清理索引（定时任务每周清理 published=true 且 published_at 超过 90 天的记录）
CREATE INDEX idx_outbox_published_at ON domain_event_outbox (published_at) 
    WHERE published = TRUE;
```

**事件发布实现：**

```java
/**
 * 领域事件发布接口
 */
public interface DomainEventPublisher {
    void publish(DomainEvent event);
}

/**
 * Outbox实现 — 事件写入数据库，和业务操作同事务
 */
@Component
public class OutboxDomainEventPublisher implements DomainEventPublisher {
    
    @Autowired
    private DomainEventOutboxRepository outboxRepo;
    
    @Override
    public void publish(DomainEvent event) {
        // 不直接发MQ或Spring Event，而是写入Outbox表
        // 因为调用方在@Transactional中，此处写入和业务操作同事务
        outboxRepo.save(DomainEventOutbox.builder()
            .eventId(event.getEventId())
            .eventType(event.getEventType())
            .aggregateType(event.getAggregateType())
            .aggregateId(event.getAggregateId())
            .payload(JsonUtils.toJson(event.getPayload()))
            .published(false)
            .build());
    }
}
```

**事件投递（Relay）实现：**

```java
/**
 * Outbox事件投递器 — 定时扫描未发布事件，投递到Spring Event Bus
 * 后期拆微服务时，改为投递到RabbitMQ
 */
@Component
public class OutboxEventRelay {

    @Autowired
    private DomainEventOutboxRepository outboxRepo;
    
    @Autowired
    private ApplicationEventPublisher springEventPublisher;

    /**
     * 每秒扫描一次未发布事件
     */
    @Scheduled(fixedDelay = 1000)
    @Transactional
    public void relayEvents() {
        List<DomainEventOutbox> events = outboxRepo.findTop50ByPublishedFalseOrderByCreatedAtAsc();
        
        for (DomainEventOutbox outbox : events) {
            try {
                // 投递到Spring ApplicationEvent
                springEventPublisher.publishEvent(
                    outbox.toDomainEvent());
                
                // 标记已发布
                outbox.setPublished(true);
                outbox.setPublishedAt(LocalDateTime.now());
                outboxRepo.save(outbox);
                
            } catch (Exception e) {
                log.error("事件投递失败: eventId={}, type={}", 
                    outbox.getEventId(), outbox.getEventType(), e);
                // 不标记published，下次重试
                // 可加重试计数，超过阈值告警
            }
        }
    }
}
```

**保证效果：**

| 情况 | 结果 |
|------|------|
| 业务操作成功 + 事件写入Outbox成功 | ✅ 定时任务投递事件 |
| 业务操作成功 + JVM崩溃（Outbox已提交） | ✅ 重启后定时任务继续投递 |
| 业务操作成功 + Outbox写入失败 | ✅ 整个事务回滚，业务数据也不保存 |
| 事件投递失败 | ✅ 下次重试，不丢失 |
| 事件投递成功 + 下游处理失败 | 由下游的幂等消费保障 |

**后期拆微服务的演进路径：**

```
初期：Outbox表 → 定时任务 → Spring ApplicationEvent（本地同步）

后期：Outbox表 → Debezium监听binlog → RabbitMQ → 微服务消费

改造点：只需改OutboxEventRelay的实现，业务代码不变
```

### 3.4 幂等性 — 解决C-05

**适用场景：** 前端重复提交、MQ重复消费、超时重试

#### 3.4.1 方案一：唯一业务键

利用业务数据的天然唯一性，数据库层面加约束：

```sql
-- 收货行：同一ASN行不能重复收货
ALTER TABLE warehouse_receiving_line 
ADD CONSTRAINT uk_receiving_asn_line UNIQUE (asn_line_id);

-- 编码流水号：同一规则同一重置键唯一
ALTER TABLE md_coding_sequence 
ADD CONSTRAINT uk_coding_rule_reset UNIQUE (rule_id, reset_key);

-- 库存预留：同一订单同一SKU同一仓库同一批次唯一
-- 注意：同一订单的同一SKU可能需要从多个批次预留，batch_no 参与唯一约束
ALTER TABLE inventory_allocation 
ADD CONSTRAINT uk_allocation_order_sku_wh_batch 
UNIQUE (order_type, order_id, COALESCE(sku_id, 0), COALESCE(material_id, 0), warehouse_id, COALESCE(batch_no, ''));

-- 库存操作流水：不设置来源维度的唯一约束
-- 原因：一个出库单包含多个SKU，每个SKU一条操作记录，source_type+source_id+operation_type 会重复
-- 防重复依靠幂等令牌 + 状态机前置校验
```

**效果：** 即使重复提交，数据库唯一约束拒绝第二次写入。

#### 3.4.2 方案二：幂等令牌（Idempotency Key）

前端每次操作携带一个UUID令牌，后端记录已处理的令牌：

```java
/**
 * 幂等令牌拦截器
 */
@Component
public class IdempotencyInterceptor implements HandlerInterceptor {
    
    @Autowired
    private IdempotencyKeyRepository keyRepo;

    @Override
    public boolean preHandle(HttpServletRequest request, 
                             HttpServletResponse response, 
                             Object handler) {
        String key = request.getHeader("X-Idempotency-Key");
        if (key == null) {
            return true; // 非幂等操作，放行
        }
        
        // 检查是否已处理（Redis SETNX，高效）
        Boolean isNew = redisTemplate.opsForValue()
            .setIfAbsent("idempotency:" + key, "1", Duration.ofHours(24));
        
        if (Boolean.FALSE.equals(isNew)) {
            // 已处理过，返回上次的结果
            String cachedResponse = redisTemplate.opsForValue()
                .get("idempotency:response:" + key);
            response.setContentType("application/json");
            response.getWriter().write(cachedResponse);
            return false;
        }
        
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, 
                                HttpServletResponse response, 
                                Object handler, Exception ex) {
        String key = request.getHeader("X-Idempotency-Key");
        if (key != null && ex == null) {
            // 缓存成功响应
            String responseBody =getResponseContent(response);
            redisTemplate.opsForValue()
                .set("idempotency:response:" + key, responseBody, Duration.ofHours(24));
        }
    }
}
```

**前端集成：**

```javascript
// 每个写操作自动生成幂等令牌
async function submitOrder(orderData) {
    const idempotencyKey = crypto.randomUUID();
    
    const response = await fetch('/api/sales-orders', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'X-Idempotency-Key': idempotencyKey,
        },
        body: JSON.stringify(orderData),
    });
    
    return response.json();
}

// 重试时使用同一个key
async function submitOrderWithRetry(orderData, maxRetry = 3) {
    const idempotencyKey = crypto.randomUUID(); // 一次操作生成一次
    
    for (let i = 0; i < maxRetry; i++) {
        try {
            return await submitOrderWithKey(orderData, idempotencyKey);
        } catch (e) {
            if (i === maxRetry - 1) throw e;
            await sleep(1000 * (i + 1));
        }
    }
}
```

#### 3.4.3 方案三：状态机前置校验

状态机天然防重复 — 同一状态对同一事件只能流转一次：

```java
// 订单已经是CONFIRMED，再次提交 → 状态机拒绝
stateMachine.fireEvent(CONFIRMED, SUBMIT, context);
// → 抛出 StateMachineException: "当前状态 CONFIRMED 不允许事件 SUBMIT"
```

#### 3.4.4 方案四：事件消费幂等

下游模块消费领域事件时，必须保证幂等 — 同一事件消费多次结果相同：

```java
/**
 * 采购模块订阅生产订单创建事件
 */
@Component
public class ProductionOrderCreatedSubscriber {

    @Autowired
    private MrpEngine mrpEngine;
    
    @Autowired
    private DomainEventOutboxRepository eventLogRepo;

    @EventListener
    @Transactional
    public void onProductionOrderCreated(ProductionOrderCreatedEvent event) {
        // 幂等校验：是否已处理过此事件
        if (eventLogRepo.existsByEventId(event.getEventId())) {
            log.info("事件已处理，跳过: {}", event.getEventId());
            return;
        }
        
        // 记录事件ID（幂等依据）
        eventLogRepo.save(EventLog.of(event.getEventId(), event.getEventType()));
        
        // 执行MRP计算
        mrpEngine.calculateByProductionOrder(event.getAggregateId());
    }
}
```

### 3.5 编码流水号原子递增 — 解决C-07

**方案：数据库行级锁 + 序列表**

```java
/**
 * 编码流水号原子递增
 */
public Long nextSequenceValue(Long ruleId, String resetKey) {
    // 行级锁：同一规则同一重置键，只有一个事务能操作
    CodingSequence seq = sequenceRepo.findByRuleIdAndResetKeyForUpdate(
        ruleId, resetKey);
    
    if (seq == null) {
        // 首次，创建序列行
        seq = new CodingSequence(ruleId, resetKey, 0L);
        sequenceRepo.save(seq);
    }
    
    seq.setCurrentValue(seq.getCurrentValue() + 1);
    sequenceRepo.save(seq);
    
    return seq.getCurrentValue();
}
```

### 3.6 跨仓库调拨一致性 — 解决C-08

**方案：三步操作，每步独立事务**

```java
/**
 * 调拨确认 — 源仓出库
 */
@Transactional
public void confirmTransferOut(Long transferId) {
    TransferOrder transfer = transferRepo.findById(transferId);
    
    // Step 1: 更新调拨单状态
    transfer.setStatus(IN_TRANSIT);
    transferRepo.save(transfer);
    
    // Step 2: 源仓扣减可用库存
    for (TransferLine line : transfer.getLines()) {
        inventoryDomainService.changeInventoryWithRetry(
            InventoryChangeRequest.builder()
                .operationType(OperationType.OUTBOUND_TRANSFER)
                .skuId(line.getSkuId())
                .warehouseId(transfer.getFromWarehouseId())
                .quantity(line.getQuantity())
                .build()
        );
    }
    
    // Step 3: 事件Outbox
    outboxRepo.save(DomainEvent.of("TransferOutboundCompleted", transferId, ...));
}

/**
 * 调拨到达 — 目标仓入库
 */
@Transactional
public void confirmTransferIn(Long transferId) {
    TransferOrder transfer = transferRepo.findById(transferId);
    
    // Step 1: 更新调拨单状态
    transfer.setStatus(COMPLETED);
    transferRepo.save(transfer);
    
    // Step 2: 目标仓增加可用库存
    for (TransferLine line : transfer.getLines()) {
        inventoryDomainService.changeInventoryWithRetry(
            InventoryChangeRequest.builder()
                .operationType(OperationType.INBOUND_TRANSFER)
                .skuId(line.getSkuId())
                .warehouseId(transfer.getToWarehouseId())
                .quantity(line.getQuantity())
                .build()
        );
    }
    
    // Step 3: 事件Outbox
    outboxRepo.save(DomainEvent.of("TransferInboundCompleted", transferId, ...));
}
```

**中间态（IN_TRANSIT）的保障：**

- 调拨单状态为IN_TRANSIT时，系统有明确的"在途"记录
- 定时任务扫描超时未到达的调拨单（超过3天仍为IN_TRANSIT），自动告警
- 超过7天仍未处理，调拨单自动标记为"异常"状态，由仓库主管决定是确认到达还是取消调拨（取消时源仓库存加回）
- 超时阈值写入系统配置表（key: transfer.timeout.alert.days / transfer.timeout.exception.days）

## 四、第二层：检测性保障（事中）

### 4.1 库存操作流水 before/after

每次库存变更记录变更前后的数量，可校验可对账：

```sql
-- inventory_operation 表的关键字段
operation_type     VARCHAR(32)   -- 操作类型
quantity           DECIMAL(12,2) -- 操作数量
available_before   DECIMAL(12,2) -- 变更前可用
available_after    DECIMAL(12,2) -- 变更后可用
locked_before      DECIMAL(12,2) -- 变更前锁定
locked_after       DECIMAL(12,2) -- 变更后锁定
qc_before          DECIMAL(12,2) -- 变更前质检
qc_after           DECIMAL(12,2) -- 变更后质检
total_before       DECIMAL(12,2) -- 变更前实际库存
total_after        DECIMAL(12,2) -- 变更后实际库存
```

**校验规则：**

```
ALLOCATE:      available_after = available_before - quantity
               locked_after    = locked_before + quantity
               total_after     = total_before (不变)

OUTBOUND_SALES: locked_after   = locked_before - quantity
               total_after     = total_before - quantity

INBOUND_PRODUCTION: available_after = available_before + quantity
                   total_after     = total_before + quantity
```

日终对账时，如果 `available_after - available_before ≠ 预期变动量`，说明数据异常。

## 五、第三层：兜底性保障（事后）

### 5.1 日终库存对账

每天凌晨自动执行，校验操作流水与库存余额的一致性：

```sql
-- 日终对账：操作流水汇总 vs 库存余额变动
-- 如果不一致，写入对账异常表

WITH daily_ops AS (
    SELECT 
        sku_id,
        warehouse_id,
        SUM(CASE WHEN operation_type = 'INBOUND_PRODUCTION' THEN quantity ELSE 0 END) AS inbound_prod,
        SUM(CASE WHEN operation_type = 'INBOUND_PURCHASE' THEN quantity ELSE 0 END) AS inbound_purch,
        SUM(CASE WHEN operation_type = 'OUTBOUND_SALES' THEN quantity ELSE 0 END) AS outbound_sales,
        SUM(CASE WHEN operation_type = 'OUTBOUND_MATERIAL' THEN quantity ELSE 0 END) AS outbound_mat,
        SUM(CASE WHEN operation_type = 'ALLOCATE' THEN 0 ELSE 0 END) AS allocated,
        SUM(CASE WHEN operation_type = 'ADJUST_GAIN' THEN quantity ELSE 0 END) AS adjust_gain,
        SUM(CASE WHEN operation_type = 'ADJUST_LOSS' THEN quantity ELSE 0 END) AS adjust_loss
    FROM inventory_operation
    WHERE operated_at >= CURRENT_DATE
      AND operated_at < CURRENT_DATE + INTERVAL '1 day'
    GROUP BY sku_id, warehouse_id
),
current_inv AS (
    SELECT 
        sku_id, warehouse_id,
        available_qty, locked_qty, qc_qty, total_qty
    FROM inventory_sku
    WHERE deleted = false
)
SELECT 
    ops.sku_id,
    ops.warehouse_id,
    -- 流水推算的今日变动
    (ops.inbound_prod + ops.inbound_purch + ops.adjust_gain 
     - ops.outbound_sales - ops.outbound_mat - ops.adjust_loss) AS ops_net_change,
    -- 当前库存余额（与昨日余额的差值需匹配）
    inv.total_qty AS current_total
FROM daily_ops ops
JOIN current_inv inv ON ops.sku_id = inv.sku_id AND ops.warehouse_id = inv.warehouse_id;
```

**对账异常处理：**

```java
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨2点

            "库存对账异常", 
            String.format("SKU %d 仓库 %d 库存差异 %d", 
                diff.getSkuId(), diff.getWarehouseId(), diff.getDiffQty()));
    }
}
```

## 九、后期演进：微服务下的一致性变化

当前方案基于模块化单体（同库），后期拆微服务后（异库），一致性方案的变化：

| 保障手段 | 模块化单体 | 微服务 |
|---------|-----------|--------|
| 跨模块事务 | @Transactional（强一致） | Outbox + MQ + 幂等消费（最终一致） |
| 事件投递 | Outbox → 定时任务 → Spring Event | Outbox → Debezium → RabbitMQ |
| 库存扣减 | 乐观锁（<20人并发无需Redis） | Redis + 异步落库 + 日终对账 |
| 编码流水号 | 数据库行级锁 | Redis INCR + 定期持久化 |
| 对账 | 库内SQL | 跨服务数据聚合 + 对账微服务 |

**核心变化：** 跨模块操作从"强一致"变为"最终一致"，依赖Outbox+MQ+幂等消费的三件套。但初期在模块化单体下不需要承担这个复杂度，可以等业务真正需要拆分时再演进。
