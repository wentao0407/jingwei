# 经纬项目 — 开发任务分解与验收标准

> 版本：v1.0  
> 日期：2026-04-30  
> 原则：每个任务 = 一次 AI 对话的工作量，完成后 AI 按验收标准自测自审

---

## 如何使用本文档

1. **按编号顺序开发**，前置依赖未完成时不要跳步
2. 每次对话开始时，告诉 AI 当前任务编号和名称
3. AI 完成编码后，要求 AI **逐条对照验收标准自查**
4. 自查通过后，更新 CLAUDE.md 中的已完成模块清单
5. 如自查未通过，让 AI 修复后重新自查

**验收标准分类说明：**
- 【功能】= 业务功能是否正确实现
- 【代码】= 代码质量和规范是否达标
- 【数据】= 数据库表/迁移是否正确
- 【测试】= 单元测试是否覆盖关键路径
- 【集成】= 与其他模块的集成是否正确

### T-02 公共模块 — 统一异常处理与响应封装

**参考文档：** CLAUDE.md 开发规范

**实现内容：**
1. `R<T>` 统一响应封装类（code, message, data）
2. `BizException` 业务异常类（错误码 + 中文提示）
3. `GlobalExceptionHandler` 全局异常处理（BizException → 业务错误响应，其他 → 500）
4. 错误码枚举 `ErrorCode`（按模块分段，如 10xxx=公共, 20xxx=基础数据, 30xxx=订单...）
5. 参数校验异常统一拦截（MethodArgumentNotValidException → 400 + 字段错误列表）

**验收标准：**

【功能】
- [ ] Controller 抛出 BizException 时，前端收到统一格式的错误响应（含错误码和中文提示）
- [ ] Controller 参数校验失败时，前端收到 400 + 各字段错误信息
- [ ] 未捕获异常时，前端收到 500 + "系统异常"（不暴露堆栈）

【代码】
- [ ] R 类使用泛型，不支持裸 R（必须 R<T>）
- [ ] BizException 携带错误码（字符串或数字），不只是消息
- [ ] GlobalExceptionHandler 使用 @RestControllerAdvice
- [ ] 所有类有 JavaDoc 注释

【测试】
- [ ] 单元测试：抛出 BizException → 响应包含正确的 code 和 message
- [ ] 单元测试：参数校验失败 → 响应包含 fieldErrors 列表

### T-03 公共模块 — 审计字段基类与乐观锁基类

**参考文档：** 01-基础数据模块-详细设计.md 第十一节

**实现内容：**
1. `BaseEntity` 基类：id, createdBy, createdAt, updatedBy, updatedAt, deleted, version
2. MyBatis-Plus `MetaObjectHandler` 自动填充 createdAt/updatedAt/createdBy/updatedBy
3. 逻辑删除配置：MyBatis-Plus 全局 `@TableLogic`（deleted 字段）
4. 乐观锁配置：MyBatis-Plus `OptimisticLockerInnerInterceptor`
5. 所有查询默认加 `WHERE deleted = false`（MyBatis-Plus 全局配置）

**验收标准：**

【功能】
- [ ] 新增记录时，createdAt/createdBy 自动填充
- [ ] 更新记录时，updatedAt/updatedBy 自动填充
- [ ] 删除操作执行 UPDATE SET deleted=true，不执行 DELETE
- [ ] 查询操作自动过滤 deleted=true 的记录
- [ ] 并发更新同一条记录时，后提交的事务抛出 OptimisticLockException

【代码】
- [ ] BaseEntity 使用 @Getter/@Setter，不用 @Data
- [ ] version 字段标注 @Version
- [ ] deleted 字段默认值为 false
- [ ] MetaObjectHandler 从 SecurityContext 或 ThreadLocal 获取当前用户ID

【测试】
- [ ] 单元测试：插入记录 → createdAt/createdBy 不为空
- [ ] 单元测试：更新记录 → updatedAt 晚于 createdAt
- [ ] 单元测试：逻辑删除 → 查询不到，但数据库中记录存在且 deleted=true
- [ ] 单元测试：乐观锁冲突 → 第二次更新失败

### T-05 系统管理 — JWT 认证与登录

**实现内容：**
1. 登录接口：用户名+密码 → JWT Token
2. JWT 工具类：生成、解析、验证
3. Spring Security 配置：JWT Filter、白名单路径
4. Token 有效期 8 小时
5. 登录成功后返回用户信息 + Token
6. 从 Token 中获取当前用户ID（供审计字段使用）

**验收标准：**

【功能】
- [ ] 正确的用户名密码 → 登录成功，返回 Token
- [ ] 错误的密码 → 返回"用户名或密码错误"
- [ ] 停用用户 → 拒绝登录
- [ ] 携带有效 Token 访问受保护接口 → 正常返回
- [ ] 不携带 Token 或携带无效 Token → 返回 401
- [ ] Token 过期 → 返回 401

【代码】
- [ ] 密码校验使用 BCryptPasswordEncoder.matches()
- [ ] JWT 密钥配置在 application.yml 中
- [ ] 白名单路径至少包含：/api/auth/login, /actuator/health

【测试】
- [ ] 单元测试：登录成功 → Token 可解析出用户ID
- [ ] 单元测试：Token 过期 → 验证失败

**参考文档：** 01-基础数据模块-详细设计.md 第六节

## 自测与审查 Checklist

每完成一个任务后，让 AI 执行以下自审流程：

### A. 代码规范审查
- [ ] 所有类有 JavaDoc 注释
- [ ] Controller 不含业务逻辑
- [ ] 入参使用 DTO，不直接用实体类
- [ ] 所有接口使用 POST（除文件上传外）
- [ ] 表名以 t_ 开头
- [ ] 实体类用 @Getter/@Setter，不用 @Data
- [ ] 异常使用 BizException，不用 RuntimeException
- [ ] 日志使用 @Slf4j

### B. 业务规则审查
- [ ] 核心业务逻辑在 DomainService 中
- [ ] 事务边界正确（@Transactional 在 ApplicationService）
- [ ] 乐观锁字段 version 存在且生效
- [ ] 软删除（deleted 字段）生效
- [ ] 审计字段自动填充

### C. 数据一致性审查
- [ ] 跨模块操作在同一事务中
- [ ] 领域事件通过 Outbox 发送
- [ ] 库存变更走 changeInventory 统一入口
- [ ] 关键操作写变更日志

### D. 编译与测试
- [ ] `mvn compile` 通过
- [ ] `mvn test` 通过
- [ ] Flyway 迁移脚本可执行
