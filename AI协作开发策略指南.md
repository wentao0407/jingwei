# 经纬项目 — AI 辅助开发协作策略指南

> 版本：v1.0  
> 日期：2026-04-30  
> 目的：解决"项目复杂、分阶段开发、新对话 AI 失忆"三大问题

---

## 一、你面临的核心问题

经纬项目有 **7 大业务模块、40+ 功能需求、10 个限界上下文**，涵盖状态机、MRP 引擎、四类库存、波次管理等复杂业务逻辑。一次对话不可能完成所有开发，但每次开新对话 AI 就从零开始——这是最核心的矛盾。

解决这个矛盾的关键思路：**把 AI 当成一个有文档阅读能力的、但记忆力只有一次对话的新员工。你要做的不是给 AI 记忆，而是给 AI 一套它每次都能读懂的"项目说明书"。**

---

## 二、三层上下文传递机制

### 2.1 第一层：CLAUDE.md（最重要！）

CLAUDE.md 是项目根目录下的一个特殊文件，**AI 每次打开对话都会自动读取它**。这是你解决"AI 失忆"问题的第一道防线。

CLAUDE.md 应该包含：

```
1. 项目概述（一段话说清这是什么项目）
2. 技术栈（AI 不用猜用什么框架）
3. 代码结构（每个包干什么）
4. 已完成的模块（AI 知道什么不用做）
5. 当前开发阶段（AI 知道该做什么）
6. 开发规范（命名、分层、异常处理等约定）
7. 关键设计决策（为什么不用 Spring StateMachine、为什么用 JSONB 等）
```

**使用策略：**

- 每完成一个模块，更新 CLAUDE.md 中的"已完成"部分
- 每开始一个新阶段，更新 CLAUDE.md 中的"当前开发阶段"
- CLAUDE.md 不需要写太多，但要写关键的——AI 可以读代码，它需要的是"方向"而不是"细节"

### 2.2 第二层：outputs 目录下的设计文档（已有）

你已经做了非常好的设计文档——7 份 PRD + 7 份详细设计 + 1 份架构图。这些文档是 AI 最重要的参考。

**使用策略：**

- 每次开始新对话时，告诉 AI 去读哪些文档。例如："请先阅读 outputs/01-基础数据模块-详细设计.md 和 outputs/PRD-MD-基础数据模块需求规格.md，然后实现物料主数据管理"
- 不要一次性让 AI 读所有文档——上下文窗口有限，只读当前任务相关的文档
- 如果某个功能的细节在设计文档中已经写得很清楚（比如表结构、API 设计），直接引用："按照 01-基础数据模块-详细设计.md 第二节的表结构实现"

### 2.3 第三层：AI 记忆系统（Memory）

Cowork 模式有一个持久化的记忆系统，AI 可以在对话中保存关键信息，下次对话时自动加载。

**使用策略：**

- 当 AI 做了重要的架构决策时，让它保存为 memory（例如："记住：库存扣减使用乐观锁而非 Redis，因为并发量 <20"）
- 当你纠正 AI 的做法时，让它保存 feedback（例如："记住：不要用 Lombok 的 @Data，用 @Getter/@Setter 代替"）
- 不要保存代码细节（代码本身就在文件里），只保存"为什么这样做"的决策

## 三、分阶段开发路线图

根据你的 PRD 交付里程碑和模块依赖关系，建议按以下顺序开发。**每个阶段作为一个独立的开发周期，完成后再进入下一阶段。**

### 第一阶段：项目骨架 + 基础数据模块（8-10 周）

**为什么从这里开始：** 基础数据是所有业务模块的依赖，没有物料、SKU、仓库，其他模块都无法运行。

**开发顺序：**

```
Step 1: 项目骨架搭建
  - Spring Boot 3.x 项目初始化
  - 包结构按限界上下文划分（com.jingwei.master/order/procurement/inventory/warehouse/system）
  - 公共模块（common）：统一异常、统一响应、审计字段基类、乐观锁基类
  - 数据库配置（PostgreSQL + Flyway 迁移）
  - Redis 配置
  - MyBatis-Plus / JPA 配置

Step 2: 系统管理基础（用户 + 角色 + 登录）
  - 用户表、角色表、菜单表
  - JWT 认证
  - RBAC 权限（菜单级 + 按钮级）
  - 此步完成后，后续开发才有认证基础

Step 3: 编码规则引擎
  - 所有业务单据的编号都需要编码规则
  - 行级锁或 Redis INCR 保证流水号原子递增
  - 此步完成后，后续模块才能生成单据号

Step 4: 物料主数据
  - md_material 表 + JSONB 扩展属性
  - 物料分类（树形结构）
  - 属性定义管理（元数据驱动动态表单）

Step 5: SPU/SKU 管理
  - 季节/波段管理
  - 尺码组管理
  - SPU → Color-way → SKU 三层结构
  - SKU 自动交叉生成

Step 6: 供应商 + 客户档案

Step 7: 仓库/库位档案
  - 多仓库、库区→货架→层→位四级结构
```

**AI 协作提示词示例（第一阶段）：**

```
我正在开发经纬服装生产管理系统。请先阅读项目根目录的 CLAUDE.md 了解项目背景。

当前任务：搭建项目骨架。

请按照以下要求实现：
1. Spring Boot 3.x + Java 17 项目，包名 com.jingwei
2. 按限界上下文划分模块：master, order, procurement, inventory, warehouse, system, common
3. common 模块包含：统一异常处理、统一响应封装、审计字段基类、乐观锁基类
4. PostgreSQL + Flyway 数据库迁移
5. Redis 配置（Session + 缓存）
6. MyBatis-Plus 作为 ORM

详细设计参考 outputs/01-基础数据模块-详细设计.md 的表结构定义。
```

### 第二阶段：订单管理 + 采购管理（8-10 周）

**依赖：** 第一阶段完成

**开发顺序：**

```
Step 1: 状态机引擎
  - 通用状态机框架（Transition + Condition + Action）
  - 这是订单和采购共用的基础能力

Step 2: 审批引擎
  - 审批配置表 + 审批任务表
  - 单人审批 + 多人或签
  - 审批结果通过领域事件通知业务模块

Step 3: 销售订单
  - 尺码矩阵存储模型（JSONB）
  - 销售订单 CRUD + 状态机
  - 审批集成
  - 变更日志

Step 4: 生产订单
  - 生产订单 CRUD + 状态机（含跳过逻辑）
  - 销售订单 → 生产订单转化（支持合并/拆分）
  - 订单跟踪时间线

Step 5: BOM 管理
  - BOM 版本控制
  - 三种消耗类型（固定/尺码相关/按订单）
  - 尺码用量表（JSONB）

Step 6: MRP 计算引擎
  - 六步计算流程
  - REPEATABLE READ 事务隔离
  - 采购建议生成

Step 7: 采购订单
  - 采购订单 CRUD + 状态机 + 审批
  - 到货通知（ASN）
  - 部分收货支持
```

```

请按照以下要求实现：

2. 实现 Transition、Condition、Action、StateMachine、StateMachineBuilder
3. 实现销售订单状态机配置（SalesOrderStateMachineConfig）

注意：
- 状态机要线程安全，无状态，可单例使用
- 每次转移自动写变更日志（通过监听器）
```

### 4.1 开始一个新模块

```
我正在开发经纬服装生产管理系统。请先阅读项目根目录的 CLAUDE.md。

当前任务：实现 [模块名称] 模块。

请按顺序阅读以下设计文档：
1. outputs/PRD-XX-[模块]需求规格.md — 了解需求
2. outputs/0X-[模块]-详细设计.md — 了解技术设计

然后告诉我：
1. 你对需求的理解
2. 实现计划（按什么顺序开发）
3. 有没有设计文档中不明确的地方需要我确认

确认后再开始编码。
```

### 4.2 继续上次的开发

```
我正在开发经纬系统，上次我们完成了 [已完成的部分]。
请先阅读 CLAUDE.md 了解项目当前状态。

当前任务：继续实现 [当前模块] 的 [具体功能]。

参考文档：outputs/0X-[模块]-详细设计.md 第X节

请先阅读相关代码，确认当前进度，然后继续开发。
```

### 4.3 修复 Bug 或调整已有代码

```
经纬系统 [模块名称] 出现了以下问题：[问题描述]

请先阅读 CLAUDE.md 和相关代码，定位问题原因，然后修复。

注意：不要改动已有接口的签名，只修改内部实现。如果有必要改接口，先告诉我。
```

### 4.4 添加新功能到已有模块

```
经纬系统 [模块名称] 需要新增 [功能描述]。

请先阅读 CLAUDE.md 和该模块的现有代码，了解当前架构风格，
然后按照相同的设计模式实现新功能。

参考文档：outputs/0X-[模块]-详细设计.md
```

## 五、每个对话结束时的收尾动作

**这是最容易被忽视、但最关键的步骤。** 每次对话结束前，做以下三件事：

### 5.1 更新 CLAUDE.md

让 AI 更新 CLAUDE.md 中的"已完成模块"和"当前开发阶段"。例如：

```
请更新 CLAUDE.md：
1. 在"已完成模块"中添加今天完成的 [功能/模块]
2. 更新"当前开发阶段"为 [下一阶段描述]
3. 如果有关键架构决策，添加到"设计决策记录"中
```

### 5.2 保存关键决策到 Memory

```
请保存以下记忆：
- [项目] XXX 模块已完成开发，使用了 XXX 方案
- [反馈] 不要 XXX，因为 XXX
```

### 5.3 确认代码能编译运行

```
请运行 mvn compile 确认代码没有编译错误。
如果有测试，运行 mvn test 确认测试通过。
```

## 六、AI 协作的核心原则

### 6.1 一次只做一件事

不要在一个对话中同时实现多个模块。**一个对话 = 一个模块或一个功能点。** 这样即使 AI 犯错，影响范围也有限。

### 6.2 先理解，再编码

每次让 AI 先阅读设计文档和已有代码，复述它的理解，确认无误后再开始编码。这比直接让它写代码更高效——写错了再改的成本远高于先确认再写。

### 6.3 设计文档是"契约"

你的设计文档写得很详细（表结构、API、状态机、代码示例都有了），把文档当"契约"——AI 实现必须和文档一致。如果 AI 认为文档有问题，让它先提出来讨论，而不是自作主张修改。

### 6.4 让 AI 写测试

状态机、MRP 计算、库存扣减是核心业务逻辑，必须写测试。让 AI 在实现功能的同时写单元测试，这本身就是一种验证——如果 AI 能写出通过的测试，说明它的实现至少逻辑上是自洽的。

### 6.5 增量提交，小步快跑

每完成一个小功能就验证一下（编译、运行、测试），不要攒一大堆代码再验证。如果 AI 写了 500 行代码但编译不通过，排错会非常痛苦。

### 6.6 不要让 AI 重新发明轮子

如果设计文档里已经给出了代码示例（状态机引擎、审批服务等），让 AI 基于这些示例实现，而不是让它从零设计。你的文档质量很高，AI 应该是"实现者"而不是"设计者"。

## 七、常见问题与对策

### Q1：AI 每次都从零开始，重复问同样的问题怎么办？

**对策：** 把答案写进 CLAUDE.md。如果 AI 第二次问了同样的问题，让它把这个答案保存到 CLAUDE.md，下次就不问了。

### Q2：AI 实现的和设计文档不一致怎么办？

**对策：** 明确告诉 AI："请严格按照 outputs/0X-XXX.md 的设计实现，不要自行修改。" 如果确实需要修改设计，先和 AI 讨论清楚，然后更新设计文档和 CLAUDE.md。

### Q3：AI 写的代码风格不统一怎么办？

**对策：** 在 CLAUDE.md 中写明代码规范，例如：

```markdown
## 开发规范
- Controller 只做参数校验和调用 Service，不写业务逻辑
- Service 分为 DomainService（业务逻辑）和 ApplicationService（编排）
- 所有数据库操作通过 Repository 接口
- 异常使用 BizException，不要用 RuntimeException
- 日志使用 Slf4j，不要用 System.out.println
- 实体类使用 @Getter/@Setter，不用 @Data
- 数据库迁移使用 Flyway，不要手动改表
```

### Q4：一个模块太大，一次对话做不完怎么办？

**对策：** 把大模块拆成小任务。例如"库存管理模块"可以拆为：

1. 对话1：库存表结构 + Repository + 基本 CRUD
2. 对话2：四类库存的状态转换逻辑 + 乐观锁
3. 对话3：库存预留与释放
4. 对话4：库存操作流水
5. 对话5：入库/出库单管理

每次对话结束时更新 CLAUDE.md，下次对话开场让 AI 先读 CLAUDE.md 和相关代码。

### Q5：AI 生成的代码有隐藏 Bug 怎么办？

**对策：** 
- 让 AI 为核心逻辑写单元测试（尤其是状态机、库存扣减、MRP 计算）
- 手动 review 关键代码路径
- 让 AI 做 code review："请审查以下代码，找出潜在的并发问题、数据一致性问题、边界条件遗漏"

### Q6：前端和后端分开开发还是一起开发？

**对策：** 建议先完成后端 API，再让 AI 实现前端页面。理由：

- 后端是业务逻辑的核心，先把逻辑做对
- 后端 API 定义清楚后，前端就是"对接接口"
- 一个对话中同时写前后端容易混乱

```markdown
技术栈：Java 17 + Spring Boot 3.x + PostgreSQL 15 + Redis 7 + React 18 + Ant Design Pro

## 代码结构
com.jingwei

├── order/           # 订单管理（销售订单、生产订单、退货）
├── procurement/     # 采购管理（BOM、MRP、采购订单、到货检验）

├── approval/        # 审批引擎
├── notification/    # 通知中心
├── cost/            # 成本核算
└── system/          # 系统管理（用户、角色、权限、日志）

## 当前开发阶段
[更新此行，例如：第一阶段 - 项目骨架搭建中]

## 已完成模块
[每完成一个模块，在此添加，例如：]
- [x] 公共模块（common）
- [x] 系统管理 - 用户/角色/登录
- [ ] 基础数据 - 物料主数据
- [ ] ...

## 设计决策记录
- 不用 Spring StateMachine，自研轻量状态机（转移表+策略模式）
- 物料属性用 JSONB 扩展字段，不按类型建表
- 订单行按 Color-way 存储数量矩阵（JSONB），不按 SKU 逐行存
- 库存扣减用数据库乐观锁（<20人并发），不引入 Redis 原子操作
- 跨模块通信用 Outbox + Spring Event，不引入 RabbitMQ
- 不引入 Activiti/Flowable，自研轻量审批引擎

## 开发规范
- Controller 只做参数校验和调用 Service
- Service 分 DomainService（业务逻辑）和 ApplicationService（编排）
- 所有数据库操作通过 Repository 接口
- 异常使用 BizException
- 实体类用 @Getter/@Setter，不用 @Data
- 数据库迁移用 Flyway
- 所有业务数据软删除（deleted 字段）
- 所有实体有 created_by/created_at/updated_by/updated_at/version 审计字段
```

## 十、详细任务分解与验收标准

上面第四~五节给出了分阶段的概要，但对于实际开发来说还不够细。我已将所有模块拆解为 **43 个原子任务**，每个任务 = 一次 AI 对话的工作量，并配有多维度验收标准。

详见：**开发任务分解与验收标准.md**

该文档的核心设计：

1. **每个任务都有唯一编号**（T-01 到 T-43），按依赖顺序排列，不可跳步
2. **每个任务明确标注参考文档**，AI 开工前就知道该读哪些设计文档
3. **验收标准分五类**：【功能】业务逻辑是否正确、【代码】代码规范是否达标、【数据】数据库迁移是否正确、【测试】关键路径是否有单元测试、【集成】跨模块交互是否正确
4. **文末附带通用自审 Checklist**（代码规范/业务规则/数据一致性/编译测试四类），每完成一个任务都让 AI 逐条过一遍

**使用方式：**

```
每次对话的开场白示例：

"我正在开发经纬系统。请先阅读 CLAUDE.md。
当前任务：T-09 基础数据 — 物料主数据管理。
参考文档：01-基础数据模块-详细设计.md 第二节, PRD-MD US-MD-001。
验收标准：见 开发任务分解与验收标准.md T-09 部分。
完成后请逐条对照验收标准自查，自查通过后再告诉我。"
```
