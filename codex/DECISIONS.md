# 经纬项目技术决策记录

> 用途：记录未来 AI 会话必须遵守的架构和产品决策。除非用户明确要求，否则不要随意推翻。
> 更新时间：2026-04-30。

## ADR-001：初期采用模块化单体

决策：

- 使用 Spring Boot 3 模块化单体。
- 所有模块初期共享一个 PostgreSQL 数据库。

原因：

- 初期并发小于 20 人。
- 当前复杂度主要是业务复杂度，不是部署规模复杂度。
- 单库事务更容易保证订单、库存、采购之间的一致性。
- 通过清晰模块边界和 Outbox，为后期微服务演进预留空间。

影响：

- 代码必须严格按限界上下文拆包。
- 模块之间优先通过领域事件协作。
- 直接跨模块调用必须谨慎、明确、可测试。

## ADR-002：初期不用 Spring Cloud Gateway

决策：

- 使用 Nginx 托管静态资源、处理 HTTPS、反向代理到 Spring Boot。

原因：

- 当前规模不需要网关层。
- Spring Cloud Gateway 会增加不必要的部署和调试成本。

## ADR-003：初期不用 RabbitMQ

决策：

- 初期使用 Spring ApplicationEvent + 事务性 Outbox。
- 后期拆微服务或外部集成复杂后，再演进到 Debezium + RabbitMQ。

原因：

- 模块化单体不需要一开始引入 MQ。
- Outbox 已能解决事件丢失问题。
- 后期只需替换 Relay 实现，业务代码不应大改。

## ADR-004：初期不用 MinIO

决策：

- 初期文件保存在本地文件系统。

原因：

- 文件量预计较小。
- 对象存储会增加运维复杂度。

## ADR-005：使用 PostgreSQL 15+ 和 JSONB

决策：

- PostgreSQL 是主数据库。
- JSONB 用于物料扩展属性、尺码矩阵、质检结果、MRP 明细、审批角色、标签数据等灵活结构。

原则：

- 高频筛选和关联字段必须保留为普通列。
- JSONB 不应成为重要关系字段的垃圾桶。
- 只有真实查询需要时才加 JSONB GIN 索引。

## ADR-006：成品走 SPU / 颜色款 / SKU，不进 `md_material`

决策：

- `md_material` 只包含 FABRIC、TRIM、PACKAGING。
- 成品由 `md_spu`、`md_color_way`、`md_sku` 表达。

原因：

- 成品按颜色尺码参与订单、库存、出库。
- 成品的款式层级和物料属性生命周期不同。

## ADR-007：销售订单行按颜色款存，数量用尺码矩阵 JSONB

决策：

- `order_sales_line` 一行表示一个颜色款。
- `size_matrix` JSONB 保存各尺码数量。
- 不采用“一行一个 SKU”的订单行模型。

原因：

- 服装用户按款式、颜色、尺码矩阵录入订单。
- 这个模型天然匹配前端矩阵。
- 订单阅读和颜色维度统计更直观。

影响：

- 代码中必须封装 `SizeMatrix` 值对象。
- 库存匹配和 MRP 需要 SKU 级汇总视图或函数。

## ADR-008：销售订单与生产订单是多对多

决策：

- 使用 `order_production_source` 连接生产订单行和销售订单行。

原因：

- 一张销售订单可以拆成多张生产订单。
- 多张销售订单可以合并成一张生产订单。
- 备货生产可以没有销售订单来源。

## ADR-009：生产订单行有独立状态

决策：

- `order_production_line.status` 表示每行生产进度。
- 生产订单主表状态由所有行中最滞后的状态决定。

原因：

- 同一生产订单内，不同颜色款可能处于不同生产阶段。

## ADR-010：使用轻量自研状态机

决策：

- 使用转移表 + 策略模式实现自研状态机。
- 初期不用 Spring StateMachine。
- 核心状态拓扑不放数据库配置。

原因：

- 当前状态流转以线性为主，少量条件分支。
- Spring StateMachine 功能过重、调试成本高。
- 状态规则属于核心业务逻辑，应通过代码评审、测试和发布控制。

## ADR-011：使用轻量审批引擎

决策：

- 支持 SINGLE 和 OR_SIGN。
- 初期不引入 Activiti、Flowable 等重型流程引擎。

原因：

- 当前不需要复杂条件分支、会签、加签、转签。

## ADR-012：库存写操作必须经过统一领域服务

决策：

- 库存数量只能通过 `InventoryDomainService.changeInventory` 或等价统一入口修改。

原因：

- 库存是最高风险的一致性领域。
- 每次变更都必须统一校验、加锁、写流水、发事件。

影响：

- 禁止业务代码直接 SQL 更新库存数量。
- 只有明确批准的数据修复脚本可以例外。

## ADR-013：库存变更必须写 before/after 流水

决策：

- `inventory_operation` 记录操作类型、数量、来源、可用/锁定/质检/实际库存的变更前后值。

原因：

- 支持追溯、审计、对账、问题排查。

影响：

- `inventory_operation` 应按月分区。
- 库存流水不可修改、不可删除。

## ADR-014：初期库存并发使用数据库乐观锁

决策：

- 使用 `version` 乐观锁和冲突重试。
- Redis 原子扣减只作为未来热点 SKU 降级方案。

原因：

- 当前并发规模较小。
- 数据库方案更简单、更容易对账。

## ADR-015：领域事件使用事务性 Outbox

决策：

- 业务事务内写 `domain_event_outbox`。
- Relay 将未发布事件投递到 Spring ApplicationEvent。

原因：

- 防止业务数据已提交但事件丢失。

影响：

- 事件消费者必须幂等。
- Relay 失败不能丢事件。

## ADR-016：重要写接口使用幂等令牌

决策：

- 重要写接口支持 `X-Idempotency-Key`。

原因：

- 订单、仓库、扫码作业容易出现重复点击、网络重试、重复提交。

## ADR-017：编码流水号初期使用数据库行级锁

决策：

- `md_coding_sequence` 使用 `SELECT ... FOR UPDATE` 或等价行锁递增。
- Redis INCR 只作为后期高并发备用。

原因：

- 简单可靠，避免单据号重复。

## ADR-018：MRP 结果是采购建议，不直接改库存

决策：

- MRP 只生成建议结果、来源明细和计算过程。
- 采购员审核后再转采购订单。

原因：

- MRP 可接受快照数据，采购决策需要人工审核。

## ADR-019：采购定义质检标准，仓库执行质检

决策：

- 采购模块维护质检标准和 ASN 数据。
- 仓库模块执行质检作业。
- 仓库发布质检事件，采购回写 ASN 行状态和结果。

原因：

- 避免采购和仓库两边重复维护同一质检状态。

## ADR-020：同款同色面料优先同批

决策：

- 同一 SPU + 颜色款的面料采购和领料应尽量使用同一批次。

原因：

- 服装面料不同批次可能有色差，同款同色混批会导致客户退货。

## ADR-021：销售库存在发货确认时扣减

决策：

- 创建出库单不扣库存。
- 发货确认时扣锁定库存。

原因：

- 货物实际离开仓库才应减少库存。

## ADR-022：盘点期间冻结库位

决策：

- 盘点涉及的库位状态设为 `FROZEN`。
- 冻结期间禁止出入库。

原因：

- 防止盘点过程中库存变化导致差异不准。

## ADR-023：成本核算保持轻量

决策：

- 只做采购价、领料成本、生产订单物料成本、成品加权平均成本。
- 不做人工成本、制造费用、应收应付、总账。

原因：

- 财务系统不在本期范围内。
